# 石斛病虫害识别项目 Flutter 前端

## 项目说明

这是一个面向石斛种植与养护场景的 Flutter 全平台前端项目。当前工作区除了 Flutter 前端，还能确认同级存在一个 Spring Boot 后端 `../shihu-web`，真实提供登录、设备状态、LED 控制和健康检查接口。

目标平台包括：

- Android
- iOS
- macOS
- Web
- Windows
- Linux（rpm包,deb包，AppImage包，二进制文件）
- OpenHarmony / 鸿蒙

目标架构包括:

- X86_64
- ARM64
- LoongArch


平台说明：

- `macOS` 需要在 macOS 主机上完成本地构建与签名验证。
- `OpenHarmony / 鸿蒙` 已纳入工程结构和运行时适配，但实际生成 `hap` 仍依赖对应 Flutter OHOS 工具链和本地签名配置。

## 当前代码基线

当前仓库已经不是“仅文档仓库”，而是具备首轮可运行基线的 Flutter 工程。

已完成：

- Flutter 工程初始化、主题、路由、分析规则和基础目录结构
- `Splash / Login / Home / Detect / Result / History / Settings / About` 主流程页面
- `flutter_riverpod` 状态管理接入
- 登录主链路：启动恢复会话 -> 登录页 -> 路由守卫 -> 设置页退出登录
- 真实后端已确认对接的接口：`POST /api/login`、`GET /api/check-login`、`POST /api/logout`、`GET /api/status`、`POST /api/ops/led`、`GET /api/health`
- 当前登录态已按 Spring Boot `HttpSession + Cookie` 模式收口，不再把仓库文档里的 token 方案当成真实后端
- 设置页已接入服务健康检查、设备状态读取和 LED 开关控制
- 本地设置持久化、本地历史记录持久化、本地登录会话持久化
- 识别相关页面、模型和仓储扩展点已保留，便于后续接入独立识别服务
- `macOS` 与 `OpenHarmony / 鸿蒙` 平台枚举、运行时识别和基础工程预留
- 一组基础 Widget 测试与仓储测试

当前未完成：

- 当前工作区里没有找到识别服务后端源码，`/api/v1/detect/image`、`/api/v1/detect/realtime/frame` 仍不能宣称已完成本地后端联调
- 实时识别页仍是前端预留结构，尚未改造成与当前 Spring Boot 设备监测后端一致的正式页面
- 登录 Cookie 的跨平台持久化策略已补齐主链路，但仍需结合目标部署环境继续做真机验证
- 关键集成测试、日志采集和全平台构建回归

## 关键决策摘要

以下约束已经确定，后续实现不要偏离：

- 项目主题固定为 `石斛病虫害识别`，不按泛化农业 Demo 组织。
- 正式环境不允许返回假结果、伪成功值或静默降级。
- V1 首版必须保留 `本地历史记录` 能力。
- 状态管理统一使用 `flutter_riverpod`。
- 受保护页面必须经过登录守卫；当前真实后端登录使用 `HttpSession + Cookie`，不是 token 刷新模型。
- 开发 / 测试环境仅在显式传入 `USE_MOCK_AUTH=true` 时才走受控演示登录。
- 当前工作区真实存在的后端能力以登录、设备状态、LED 控制和健康检查为准；识别服务仍按独立仓储扩展点保留。
- 历史记录和设置始终走本地真实持久化。
- 平台差异统一收敛到适配层，不在页面层分叉业务逻辑。

## 下一阶段优先级

建议按以下顺序继续推进：

1. 把独立识别服务后端源码并入工作区，或明确其仓库位置，再继续收口 `Detect / Realtime` 前端链路。
2. 继续验证 `HttpSession + Cookie` 在 Android、iOS、桌面端和 Web 的实际会话恢复行为。
3. 决定 `RealtimeDetectPage` 是改造成设备实时监测页，还是等待独立识别服务接入后继续保留当前结构。
4. 补齐更多 Widget / 集成测试，并逐步完成 Android、Web、Windows、macOS、OpenHarmony 的构建验证。
