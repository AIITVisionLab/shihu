# 斛生 客户端功能设计文档

## 1. 文档目标

本文档用于定义斛生客户端当前版本的真实功能范围、页面职责、主流程和验收边界。本文档只以当前仓库代码和 `../shihu-web` 的 `origin/web` 后端源码为准，不再沿用旧规划稿中的预留能力描述。

## 2. 当前版本定位

当前版本是一个可直接用于值守与排障的跨平台客户端基线，重点是把以下能力稳定收进统一工作台：

- 账号登录、注册、会话恢复和退出
- 设备状态轮询与异常等级展示
- LED 补光控制
- 服务健康检查与运行环境管理
- 系统总览与业务背景说明

当前版本不再提供以下能力：

- 单图识别
- 识别结果页
- 识别历史记录
- 视频中心与单路视频流详情

以上模块已经从代码库和运行时入口中删除，不再作为“预留功能”保留在当前交付基线里。

## 3. 平台范围

目标平台：

- Android
- iOS
- macOS
- Web
- Windows
- Linux
- OpenHarmony / 鸿蒙

平台要求：

- 主流程和页面信息架构在所有平台保持一致
- 首页、主控台、设置页、系统总览页统一使用工作台壳层
- 桌面端使用侧边导航与更宽内容区
- 紧凑宽度下自动切换为底部导航和单列布局
- 主视觉统一使用蓝色科技风主色，辅助使用冷青高光和深蓝灰底色
- 动效、间距、卡片层级和交互反馈在各平台保持统一风格

## 4. 当前功能范围

### 4.1 已交付功能

- 启动初始化 `SplashPage`
- 登录 / 注册 `LoginPage`
- 工作台首页 `HomePage`
- 实时监控主控台 `RealtimeDetectPage`
- 运维设置 `SettingsPage`
- 系统总览 `AboutPage`
- 本地设置、记住账号、登录会话持久化

### 4.2 当前明确不在范围内的能力

- AI 识别接口接入
- 本地或远程历史记录
- 视频流播放与视频详情
- 摄像头权限、取帧和叠框渲染

说明：

- 两份英文命名协作文档仍保留在 `doc/` 下作为未来协作参考。
- 它们不属于当前代码库已交付功能，也不应影响当前前端导航、页面和文案。

## 5. 页面与职责

### 5.1 启动页 `SplashPage`

功能目标：

- 初始化本地设置、应用版本和登录态
- 避免冷启动白屏
- 按登录态跳转登录页或主控台

当前实现：

- 并行读取本地配置和会话
- 展示初始化中、失败重试两种状态
- 初始化完成后直接进入 `LoginPage` 或 `RealtimeDetectPage`

### 5.2 登录页 `LoginPage`

功能目标：

- 统一承接登录和注册
- 作为普通用户的唯一登录入口
- 作为所有受保护页面的唯一认证入口

当前实现：

- 登录字段固定为 `Username / Password / Confirm password`
- 真实接口对接 `/api/login`、`/api/register`、`/api/check-login`、`/api/logout`
- 认证方式固定为 `HttpSession + JSESSIONID`
- 认证页默认采用居中单卡片登录布局，仅在演示环境或服务地址异常时补充轻量提示条
- 开发 / 测试环境可通过 `USE_MOCK_AUTH=true` 启用本地联调登录替身
- 登录成功后直达 `RealtimeDetectPage`

验收边界：

- 不再展示识别、视频或历史记录入口
- 不再使用 Bearer Token 口径误导用户

### 5.3 工作台首页 `HomePage`

功能目标：

- 作为登录后的总览入口
- 优先展示当前设备状态
- 为值守、说明和我的页面提供统一跳转入口

当前实现：

- 展示当前设备状态与最近同步情况
- 展示设备摘要快照
- 提供“值守台 / 使用说明 / 我的”三个一级入口
- 桌面端入口卡片具备 hover 反馈，移动端自动转为单列布局

### 5.4 实时监控页 `RealtimeDetectPage`

功能目标：

- 作为当前版本的核心业务页面
- 持续同步 `/api/status`
- 承接 LED 补光控制

当前实现：

- 页面进入后自动启动轮询
- 顶部状态区只保留轮询、刷新和退出操作
- Hero 区展示设备主状态、错误信息和异常等级
- 指标区展示温度、湿度、光照、MQ2
- 控制区展示设备身份、错误码、更新时间和 LED 开关
- 状态说明区解释错误码 0 / 1 / 2 / 其他情况

验收边界：

- 页面必须在无设备状态时保持稳定骨架，不允许白屏
- 自动刷新可关闭
- LED 控制提交中不允许重复触发

### 5.5 我的页 `SettingsPage`

功能目标：

- 管理当前设备、登录账号和本机偏好
- 收纳退出登录、清除记住账号和恢复默认等用户操作

当前实现：

- 当前设备卡片展示设备名称、当前状态、最近同步和应用版本
- 当前账号卡片展示登录账号、登录模式和会话到期时间
- 本地偏好卡片支持清除记住账号和恢复默认

验收边界：

- 默认不向普通用户暴露服务地址或健康检查等后台字段
- 当前不再提供清理历史记录等遗留操作

### 5.6 使用说明页 `AboutPage`

功能目标：

- 用用户视角解释页面怎么使用
- 让总览、值守和我的三条主路径保持一致

当前实现：

- 顶部 Hero 区展示三条主路径、版本边界和核心使用原则
- 主路径区按轨道式卡片解释“总览 / 值守 / 我的”分别什么时候进、重点看什么、进去后做什么
- 协作区根据 `doc/java-video-collaboration.md` 与 `doc/java-ai-collaboration.md` 摘要说明未来视频 / AI 接入边界，但不新增当前运行时入口

## 6. 当前主流程

### 6.1 认证主流程

`SplashPage -> LoginPage -> RealtimeDetectPage -> HomePage / SettingsPage / AboutPage`

### 6.2 值守主流程

`RealtimeDetectPage -> 查看状态 -> 判断异常等级 -> 切换 LED -> 等待状态回写`

### 6.3 排障主流程

`SettingsPage -> 查看服务地址 -> 检查 /api/health -> 查看设备状态 -> 必要时退出并重新登录`

## 7. 设计与体验要求

- 保持 Material 3 信息架构和组件语义
- 页面背景、卡片、动效和导航风格必须统一为同一套深色工作台语言
- 桌面端卡片布局最多两列，手机端优先单列，避免操作页出现碎片化信息流
- 默认省略重复说明和跨页重复信息，优先保证主要操作链路清晰
- 顶层模块切换必须使用工作台导航直达，不叠加页面返回栈
- 一级导航切换必须保持已访问板块状态，不重复播放首屏入场动效
- 动效只保留必要反馈，不做炫技装饰
- 在桌面端与移动端都要保证点击路径清晰、响应流畅
- 退出登录、恢复默认设置和清除本机记住账号必须二次确认
- 不允许再把未接入后端的能力当成“占位入口”展示给用户

## 8. 后续优先级

按当前版本基线，后续优先级如下：

1. 验证 Android、iOS、Windows、macOS、OpenHarmony / 鸿蒙 的真实构建与会话恢复行为
2. 继续优化主控台和设置页在真机上的滚动、输入和反馈细节
3. 未来如需恢复 AI / 视频协作，必须先补齐对应后端源码，再重新立项，不在当前代码库内做预留实现
