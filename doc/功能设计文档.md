# 病虫害检测项目 Flutter 前端功能文档

## 1. 文档目标

本文档用于定义病虫害识别项目 Flutter 前端的功能范围、页面职责、交互流程和验收要求，作为后续开发、联调、演示和答辩的功能依据。

本文档重点回答以下问题：

- 这个前端要实现哪些功能
- 每个页面承担什么职责
- 用户在各个平台上如何完成主要操作
- 系统应当提供哪些反馈、状态和异常处理
- 当前无后端时，哪些功能可以先用 mock 方式完成

## 2. 产品概述

病虫害识别项目是一个面向农业病虫害识别场景的跨平台前端应用，支持用户通过上传图片或摄像头实时预览的方式进行识别，并查看识别结果、历史记录和基础配置。

产品目标：

- 提供统一、中文、易演示的跨平台界面
- 支持 Android、iOS、Web、Windows、Linux 全平台使用
- 在无后端阶段也能完成完整的静态演示和交互演示
- 后续可平滑对接 Java 后端识别服务

## 3. 目标用户与使用场景

### 3.1 目标用户

- 项目演示人员：用于比赛答辩、录屏、展示项目能力
- 普通使用者：上传作物图片或查看识别结果
- 开发与联调人员：配置服务地址、切换环境、验证接口返回

### 3.2 主要使用场景

- 比赛答辩现场演示病虫害识别流程
- 使用本地图片进行单图识别演示
- 使用摄像头进行实时识别观感演示
- 查看历史识别记录和结果详情
- 对接本地或局域网 Java 后端进行联调测试

## 4. 功能范围总览

本项目当前功能范围包括：

- 启动页
- 首页
- 单图识别
- 实时识别
- 结果展示
- 历史记录
- 设置
- 关于项目

全局支持能力包括：

- 全平台统一导航与交互逻辑
- 中文文案
- Material Design 3 风格界面
- 流畅动画与状态反馈
- mock 数据模式与真实接口模式切换
- JSON / HTTP / multipart 与 Java 后端联调预留

## 5. 全局功能要求

### 5.1 平台支持

目标平台：

- Android
- iOS
- Web
- Windows
- Linux

要求：

- 主流程在各平台保持一致
- 页面结构和业务逻辑尽量复用
- 只在权限、窗口尺寸、输入媒介等方面做必要适配

### 5.2 统一交互要求

- 首页入口顺序固定
- 页面返回逻辑统一
- 同类状态使用同类反馈方式
- 按钮、卡片、提示、弹窗使用统一组件风格
- 用户完成一次操作后，系统应立即给出反馈

### 5.3 动画与体验要求

- 页面切换使用统一转场动画
- 重点模块支持轻量入场动画
- 点击、加载、成功、失败必须有反馈
- 动画不影响识别页可读性和交互响应
- 桌面端与 Web 端支持 hover、focus、键盘操作等补充体验

### 5.4 网络与配置要求

- 统一使用 `HTTP / HTTPS`
- 接口数据以 `JSON` 为主
- 图片上传采用 `multipart/form-data`
- 服务地址、超时时间、日志开关集中配置
- 支持后续对接 Java 后端 REST API

## 6. 页面与功能明细

## 6.1 启动页（SplashPage）

### 功能目标

- 展示项目 Logo、名称和启动品牌感
- 完成应用初始化
- 自动跳转首页

### 当前阶段功能

- 显示 Logo / 项目名称 / 简短标语
- 延时 1 到 2 秒后进入首页

### 后续可扩展功能

- 读取本地配置
- 检查服务地址
- 检查权限状态
- 检查缓存或历史记录

### 交互要求

- 启动页动画轻量、简洁
- 不允许停留过久
- 初始化失败时给出重试或继续进入选项

## 6.2 首页（HomePage）

### 功能目标

- 作为整个应用的主入口
- 展示项目定位和主要功能
- 提供统一导航

### 页面内容

- 项目标题
- 项目简介
- 功能入口卡片或按钮
- 版本信息或状态信息

### 核心功能

- 进入单图识别页
- 进入实时识别页
- 进入历史记录页
- 进入设置页

### 交互要求

- 首页视觉完成度要高，适合展示
- 首屏信息清晰，避免空白感
- 入口按钮明显，主次分明
- 支持轻量入场动画

## 6.3 单图识别页（DetectPage）

### 功能目标

- 支持用户从相册选择图片或使用相机拍照
- 触发识别流程
- 将结果传递到结果页展示

### 当前阶段功能

- 选择图片
- 预览图片
- 点击开始识别
- 使用 mock 数据返回识别结果

### 后续联调功能

- 上传图片到后端
- 接收识别结果 JSON
- 支持加载态、失败态、重试

### 交互流程

1. 用户进入单图识别页
2. 选择图片或拍照
3. 页面展示所选图片
4. 用户点击开始识别
5. 显示识别中状态
6. 成功后跳转结果页
7. 失败时提示错误并支持重试

### 状态要求

- 未选择图片
- 已选择图片
- 识别中
- 识别成功
- 识别失败

### 异常处理

- 用户未选择图片时禁止识别
- 读取图片失败时提示重新选择
- 网络异常时提示检查服务地址或网络连接

## 6.4 实时识别页（RealtimeDetectPage）

### 功能目标

- 打开摄像头进行实时预览
- 展示实时识别观感
- 为后续实时识别能力预留入口

### 当前阶段功能

- 打开摄像头预览
- 显示假框 overlay
- 显示当前识别状态信息

### 后续联调功能

- 定时采样图像帧
- 上传帧数据或调用实时识别接口
- 返回框信息并更新 overlay

### 页面结构

- 底层：摄像头预览
- 中层：检测框 Overlay
- 顶层：状态栏、识别信息卡片、操作按钮

### 交互要求

- 识别框更新平滑
- 顶层控件不遮挡主要画面
- 页面响应要稳，避免闪烁和明显掉帧
- 当摄像头不可用时提供说明和退出方式

### 异常处理

- 摄像头权限被拒绝
- 摄像头初始化失败
- 当前平台不支持相关能力

## 6.5 结果页（ResultPage）

### 功能目标

- 展示识别结果详情
- 强化信息层级和可读性
- 支持将结果保存为历史记录

### 页面内容

- 原图或结果图
- 检测框信息
- 病虫害类别
- 置信度
- 严重程度
- 防治建议
- 保存记录按钮

### 当前阶段功能

- 展示 mock 结果
- 根据结果数据生成信息卡片
- 保存到本地历史记录

### 交互要求

- 结果信息分区清晰
- 识别完成后内容分步呈现
- 严重程度可用颜色标签强化表达
- 保存后给出明确成功反馈

### 异常处理

- 结果数据为空时展示空状态
- 保存失败时提示重试

## 6.6 历史记录页（HistoryPage）

### 功能目标

- 展示历史识别记录
- 支持查看详情

- 查看历史记录列表

### 交互要求

- 列表滚动流畅

## 6.7 设置页（SettingsPage）

## 7. 功能流程说明

### 7.1 单图识别流程

1. 用户从首页进入单图识别页
2. 选择图片或拍照
3. 页面展示图片预览
4. 用户点击开始识别
5. 前端进入加载态
6. 当前阶段由 mock 数据返回结果，联调阶段改为请求后端
7. 跳转到结果页展示详情
8. 用户可保存到历史记录

### 7.2 实时识别流程

1. 用户从首页进入实时识别页
2. 应用初始化摄像头
3. 页面展示摄像头预览
4. 当前阶段显示假框与模拟识别信息
5. 后续联调阶段按固定频率向后端请求识别结果
6. 前端实时更新检测框和结果卡片

### 7.3 历史记录查看流程

1. 用户从首页进入历史记录页
2. 页面加载本地记录
3. 用户查看列表
4. 点击任意记录进入结果详情页

### 7.4 设置修改流程

1. 用户进入设置页
2. 修改服务地址或其他配置
3. 页面执行保存
4. 保存成功后给出提示
5. 后续网络请求使用新配置

## 8. 数据与后端联调需求

### 8.1 基础通信方式

- 协议：`HTTP / HTTPS`
- 数据格式：`JSON`
- 编码：`UTF-8`
- 上传：`multipart/form-data`

### 8.2 推荐接口形式

- `POST /detect/image`：上传图片并返回识别结果
- `GET /history/list`：获取历史记录列表
- `GET /history/{id}`：获取历史详情

### 8.3 推荐返回结构

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

### 8.4 联调要求

- 前端统一由 `ApiClient` 发起请求
- 页面不直接拼接接口地址
- `baseUrl` 可在设置页修改
- 支持统一处理 401、403、404、500、超时、无网络等异常
- Web 联调时需考虑 Java 后端 CORS 配置

## 9. 当前阶段 Mock 功能要求

在无后端阶段，以下能力必须先通过 mock 完成：

- 单图识别结果
- 结果页信息卡片
- 历史记录列表与详情

- 设置页配置保存

Mock 的目的：

- 先打通完整前端流程
- 支撑比赛演示和录屏
- 后续仅替换 service 层即可切换到真实接口

- 页面切换流畅
- 历史列表滚动顺畅
- 实时识别页 overlay 更新平滑
- 低性能设备上也不出现明显卡死

### 10.2 视觉要求

- 中文界面统一
- Material Design 3 风格统一
- 首页、结果页、实时识别页重点打磨
- 配色、圆角、间距、阴影规则统一

### 10.3 可用性要求

- 主要按钮易于发现
- 错误提示清晰
- 空状态有说明
- 保存、删除、清空等操作需有反馈

### 10.4 跨平台一致性要求

- Android、iOS、Web、Windows、Linux 主流程一致
- 业务逻辑不按平台分裂
- 桌面端补充 hover、focus、键盘交互
- 平台差异控制在权限和布局适配层
