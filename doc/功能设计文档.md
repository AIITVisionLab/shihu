# 斛生 客户端功能设计文档

## 1. 文档目标

本文档用于定义斛生客户端当前版本的真实功能范围、页面职责、主流程和验收边界。本文档只以当前仓库代码和 `../shihu-web` 的 `origin/web` 后端源码为准，不再沿用旧规划稿中的预留能力描述。

## 2. 当前版本定位

当前版本是一个可直接用于值守与排障的跨平台客户端基线，重点是把以下能力稳定收进统一工作台：

- 账号登录、注册、会话恢复和退出
- 设备状态轮询与异常等级展示
- LED 补光控制
- 视频中心与软件内视频查看
- AI 巡检结果查看
- 平台日志摘要、筛选与最近事件查看
- 使用帮助与页面信息说明

当前版本不再提供以下能力：

- 单图识别
- 识别结果页
- 识别历史记录

以上模块已经从代码库和运行时入口中删除，不再作为“预留功能”保留在当前交付基线里。

## 3. 平台范围

目标平台：

- Android
- iOS
- macOS
- Web
- Windows
- Linux
- OpenHarmony / 鸿蒙

平台要求：

- 主流程和页面信息架构在所有平台保持一致
- 首页、主控台、设置页、系统总览页统一使用工作台壳层
- 桌面端使用侧边导航与更宽内容区
- 紧凑宽度下自动切换为底部导航和单列布局
- 全局品牌统一使用透明兰花图标，界面辅助装饰围绕品牌主色展开
- 主视觉统一使用 `#518463` 青松绿主色，辅以 `#A7D3B2 / #CBF2E0 / #D2C8AC / #CEBBD8`
- 背景底色统一保持偏冷雾白纸感，不再回到偏黄暖灰底
- 工作台页头统一使用与正文同宽的紧凑标题条，不再使用独立大头卡片
- 二级信息卡统一使用共享卡片语法，不允许各页面继续堆叠重复的装饰线、说明块和局部盒子
- 主体大卡和 Hero 统一以白纸底为主，颜色只保留在标签、顶线和局部强调面里
- 全站导航标签、登录辅助面板、视频入口和二级信息卡统一复用首页“环境速览”的浅松绿、雾青、亚麻、淡紫强调顺序
- 二级信息卡统一以左上短色线 + 白底浅色染面为主要视觉语法
- 用户界面不展示独立的配色说明区或品牌信息卡
- 动效、间距、卡片层级和交互反馈在各平台保持统一风格

## 4. 当前功能范围

### 4.1 已交付功能

- 启动初始化 `SplashPage`
- 登录 / 注册 `LoginPage`
- 工作台首页 `HomePage`
- 实时监控主控台 `RealtimeDetectPage`
- 运维设置 `SettingsPage`
- 视频中心 `VideoPage`
- 使用帮助 `AboutPage`
- 本地设置、记住账号、登录会话持久化

### 4.2 当前明确不在范围内的能力

- 本地或远程历史记录
- 摄像头权限、取帧和叠框渲染
- 单图识别页、识别结果页和对应历史页的恢复

说明：

- 视频中心已按两份英文命名协作文档接入，但界面层不向普通用户暴露技术接入过程。
- 当前帮助页只保留用户视角的页面说明和查看重点。

## 5. 页面与职责

### 5.1 启动页 `SplashPage`

功能目标：

- 初始化本地设置、应用版本和登录态
- 避免冷启动白屏
- 按登录态跳转登录页或主控台

当前实现：

- 并行读取本地配置和会话
- 展示启动图标区、初始化中和失败重试三类内容
- 初始化完成后直接进入 `LoginPage` 或 `RealtimeDetectPage`

### 5.2 登录页 `LoginPage`

功能目标：

- 统一承接登录和注册
- 作为普通用户的唯一登录入口
- 作为所有受保护页面的唯一认证入口

当前实现：

- 登录字段固定为 `Username / Password / Confirm password`
- 真实接口对接 `/api/login`、`/api/register`、`/api/check-login`、`/api/logout`
- 认证方式固定为 `HttpSession + JSESSIONID`
- 认证页采用桌面说明区 + 独立表单卡片、移动端单列的一体化登录布局
- 登录说明区明确“默认进入值守台”以及总览 / 视频 / 我的的职责，不再复述首页式流程口号
- 登录页提供“直接预览界面”入口，允许在后端暂不可用时写入本地预览会话并进入工作台；仅在检测到自定义服务配置时展示恢复默认提示，不直出原始服务地址
- 登录成功后直达 `RealtimeDetectPage`

验收边界：

- 不再展示识别、视频或历史记录入口
- 不再使用 Bearer Token 口径误导用户

### 5.3 工作台首页 `HomePage`

功能目标：

- 作为登录后的总览入口
- 优先展示当前设备状态
- 为当前值守状态提供一眼可读的总览入口

当前实现：

- 顶部总览区使用信号盘面式 Hero，优先展示当前设备结论、最近同步和补光状态
- 桌面端首卡统一收为“主信息区 + 固定侧区”的比例结构，窗口放大后不继续拉成长头部，正文最大宽度统一放宽到 `1360`
- 桌面端品牌和当前账号统一放在左侧导航仓，页头只保留页面标题与当前页说明
- 首页正文不再重复渲染“总览 / 值守 / 视频 / 我的”一级导航卡，模块切换统一交给左侧导航仓或底部导航
- 环境速览区集中展示设备摘要、四项核心指标和补光状态，首页以下内容只保留与 `/api/status` 直接相关的信息
- 首页以下的二级卡片统一继承总览页的色块标题、光晕和分区语义，减少页面间的风格断层

### 5.4 实时监控页 `RealtimeDetectPage`

功能目标：

- 作为当前版本的核心业务页面
- 持续同步 `/api/status`
- 承接 LED 补光控制
- 查看后端汇总后的 AI 巡检结论

当前实现：

- 页面进入后自动启动轮询
- Hero 区展示设备主状态、当前结论、最近同步与值守节奏
- 桌面端首卡统一收为“主信息区 + 固定侧区”的比例结构，侧区宽度统一按 `328-392` 自适应收口，避免全屏后首卡继续横向膨胀
- 指标区展示温度、湿度、光照、MQ2
- 控制区统一收纳设备名称、当前状态、补光状态、最近同步、LED 开关和处理建议
- 处理建议继续使用“正常 / 注意 / 严重 / 待确认”这类用户视角标签，不直接暴露错误码
- 桌面端优先使用“指标主列 + 操作说明侧列”的两栏布局，主体区与首页、设置共用同一套双栏断点与间距
- 四项环境指标统一改为矩阵式卡片，移动端优先双项网格
- 设备摘要、同步状态和 LED 补光统一收在同一张操作卡片中，不再追加独立的说明型小卡
- 值守页桌面端下半区按内容自然高度排布，不再用固定高度强行等高
- 值守页二级卡片复用总览页的浅松绿、雾青、亚麻、淡紫强调顺序，避免指标区和操作区退回偏灰白底
- AI 巡检卡直接查询 `/api/edge/ai-detections/latest` 和 `/api/edge/ai-detections/history`
- AI 巡检卡展示最新结论、最近记录和重点目标建议，但不恢复独立识别页面

验收边界：

- 页面必须在无设备状态时保持稳定骨架，不允许白屏
- 自动刷新可关闭
- LED 控制提交中不允许重复触发
- AI 接口尚未返回结果时必须展示稳定空状态，不伪造检测结论

### 5.5 我的页 `SettingsPage`

功能目标：

- 管理当前设备、登录账号和本机偏好
- 收纳退出登录、清除记住账号和恢复默认等用户操作

当前实现：

- 当前使用总览卡片统一展示设备状态、当前账号、最近同步、数据状态和补光状态
- 当前使用总览采用“设备主卡 + 信息矩阵”的控制台式布局
- 桌面端首卡统一收为“主信息区 + 固定侧区”的比例结构，不再让摘要头部长成独立大块
- 账号与本机主卡统一收纳当前账号、记住账号、退出登录、恢复默认和帮助入口
- 对不支持安全持久化登录态的平台，账号与本机主卡追加明确提示，说明关闭应用后需要重新登录
- 平台日志卡支持按关键字、事件类型和条数范围重新查询，继续保持默认展示最近 6 条
- 设置页整体保持“总览 Hero + 账号与本机主卡 + 平台日志”的紧凑工作站结构
- 设置页总览板、账号与本机主卡统一收为更紧凑的状态块与操作块，减少说明性留白

验收边界：

- 默认仍不向普通用户暴露服务地址等高风险配置项
- 平台事件只展示真实结果，不伪造运行状态
- 当前不再提供清理历史记录等遗留操作

### 5.6 视频中心 `VideoPage`

功能目标：

- 作为视频访问与在线状态核对的统一入口
- 直接承接后端下发的视频地址

当前实现：

- 调用 `GET /api/video/streams`
- 当业务后端尚未提供 `/api/video/streams` 时，按本地运行配置回退到单路公网视频网关入口
- 页面顶部优先收纳“当前主画面”，并在卡片内直接开始播放
- 桌面端首卡统一使用“主画面区 + 固定宽度控制区”的比例结构，控制区宽度跟随统一令牌自适应收口，避免全屏后主卡失衡
- 其余画面作为补充入口下沉到主画面下方，并改为紧凑入口列表，不再额外套列表外层卡片
- 视频卡片展示 `aiResultForwarded` 状态，提示当前流是否已收到 AI 结果
- Android / iOS / macOS 点击画面入口时优先进入软件内播放页，由应用内嵌 WebView 承接后端返回的播放页面
- Windows / Linux 点击画面入口时优先进入软件内播放页，由原生播放器直接拉取 go2rtc 媒体流
- Web 点击画面入口时优先进入应用内播放页，由浏览器原生 `iframe` 直接承接跨源播放地址
- 当前暂不支持软件内播放的平台，会在播放页明确提示平台能力范围
- 页面保留“主画面直播放卡 + 其它画面紧凑入口”两层结构，不再追加单独的查看提示卡
- 页面默认不展示地址、主机、端口等技术字段
- 移动端入口统一回到单列，桌面端保持紧凑右侧操作布局

验收边界：

- 页面获取失败时必须明确展示失败状态，不伪造成功内容
- 页面为空时不展示空壳播放器
- 媒体链路直接走播放地址，不通过业务后端代理视频字节流

### 5.7 使用帮助页 `AboutPage`

功能目标：

- 用用户视角解释页面怎么使用
- 把主路径和查看重点集中放到一页

当前实现：

- 顶部 Hero 区以导览面板形式汇总四个主入口和查看顺序
- 桌面端首卡统一收为“主信息区 + 固定侧区”的比例结构，复用与首页、值守一致的侧区宽度规则，避免帮助页头部比其它页面更重
- 主路径区按轨道式卡片解释“总览 / 值守 / 视频 / 我的”分别什么时候进、进去后做什么
- 页面把“平时重点看哪些信息”并入同一张导览卡，不再拆成第二张重复说明卡
- 页面整体收敛为导览型界面，不再保留说明文档式长段落结构
- 未登录时页头额外提供“打开界面预览”入口，进入后只加载本地样例数据，不依赖在线接口

## 6. 当前主流程

### 6.1 认证主流程

`SplashPage -> LoginPage -> RealtimeDetectPage -> HomePage / VideoPage / SettingsPage / AboutPage`

### 6.2 值守主流程

`RealtimeDetectPage -> 查看状态 -> 判断异常等级 -> 切换 LED -> 等待状态回写`

### 6.3 排障主流程

`HomePage / RealtimeDetectPage -> 查看当前状态 -> 需要时进入“我的”处理账号或本机偏好 -> 必要时退出并重新登录`

## 7. 设计与体验要求

- 保持 Material 3 信息架构和组件语义
- 透明兰花品牌图标必须在启动页、登录页和工作台导航中保持统一出现
- 页面背景、卡片、动效和导航风格必须统一为同一套浅纸感松青工作台语言
- 背景底色统一控制在偏冷浅雾白范围，避免松青主色被暖米底色带偏
- 工作台壳层统一使用悬浮式导航仓与标题控制仓，不再混用默认页面容器样式
- 装饰层只允许作为弱化背景存在，不使用大面积独立色块抢占正文视觉重心
- 桌面端卡片布局最多两列，手机端优先单列，避免操作页出现碎片化信息流
- 页面文案必须使用用户视角，不直接暴露内部设计概念和开发协作口径
- 默认省略重复说明和跨页重复信息，优先保证主要操作链路清晰
- 顶层模块切换必须使用工作台导航直达，不叠加页面返回栈
- 一级导航切换必须保持已访问板块状态，不重复播放首屏入场动效
- 动效只保留必要反馈，不做炫技装饰
- 在桌面端与移动端都要保证点击路径清晰、响应流畅
- 退出登录、恢复默认设置和清除本机记住账号必须二次确认
- 不允许再把未接入后端的能力当成“占位入口”展示给用户
- 通用卡片统一使用松绿细高光、克制深色渐变面和一致的标题层级，不允许页面各自堆一套装饰语法

## 8. 后续优先级

按当前版本基线，后续优先级如下：

1. 验证 Android、iOS、Windows、macOS、OpenHarmony / 鸿蒙 的真实构建与会话恢复行为
2. 继续优化主控台和设置页在真机上的滚动、输入和反馈细节
3. 未来如需恢复单图识别页、结果页和叠框渲染，必须重新立项，不在当前代码库里保留旧入口
