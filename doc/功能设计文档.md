# 斛生 客户端功能设计文档

## 1. 文档目标

本文档用于定义斛生客户端的当前功能范围、页面职责、用户流程和验收边界，作为后续开发、联调和文档同步的功能依据。

本文档重点回答以下问题：

- 当前代码已经实现了哪些功能
- 哪些页面已经可用，哪些只是占位入口
- 用户在各平台上能完成哪些主流程
- 哪些能力仍然处于下一阶段规划

## 2. 产品概述

本项目是一个面向石斛幼苗培育与设备管理场景的跨平台客户端应用。当前客户端已收口到设备管理主链路，并对接线上部署的 Spring Boot 设备管理后端。

当前版本重点不是“做一套静态演示稿”，而是先打通一个可直接投入值守使用的客户端基线：

- 用户可以登录并恢复上一次会话
- 用户可以查看视频流清单、单路详情和播放入口
- 用户可以查看环境、版本、平台和本地配置
- 用户可以查看设备状态、服务健康状态并控制 LED
- 用户可以查看系统总览、平台总览和运维设置
- 未接入后端支撑的识别相关能力已从当前运行时入口移除

当前版本尚未完成：

- 独立识别服务后端并入工作区
- `Detect` 与真实识别服务的正式联调
- 摄像头实时识别链路的权限、取帧和页面级渲染
- 全平台发布级构建验证

## 3. 当前版本功能范围

### 3.1 已实现功能

- 启动页初始化
- 登录页
- 首页主导航
- 视频中心
- 单路视频流详情
- 实时监控主控台
- 设置页
- 系统总览页
- 本地设置持久化

### 3.2 已预留但未完成的功能

- 独立识别服务的真实接口接入
- 单图识别、结果页和识别历史在独立识别服务并入后的重新开放
- 摄像头实时识别链路恢复为独立产品能力
- 平台运行时识别与 `OpenHarmony / 鸿蒙` 编译期平台覆盖

### 3.3 当前不提供的能力

- 假摄像头预览
- 假检测框 overlay
- 远程历史记录接口
- 服务端配置托管

说明：

- 当前工作区里真实存在的后端接口以 `/api/login`、`/api/check-login`、`/api/logout`、`/api/status`、`/api/ops/led`、`/api/health` 为准。
- 识别服务相关接口当前仍缺少本地后端源码支撑。
- 历史记录和设置页面使用本地真实持久化，不再写成“mock 列表 / mock 设置”。

## 4. 平台与环境范围

### 4.1 目标平台

- Android
- iOS
- macOS
- Web
- Windows
- Linux
- OpenHarmony / 鸿蒙

### 4.2 平台要求

- 各平台主流程入口顺序保持一致
- 页面结构和业务逻辑尽量复用
- 平台差异主要收敛到权限、文件选择、窗口尺寸和运行时判断
- `OpenHarmony / 鸿蒙` 当前以工程接入和运行时适配为主，实际构建依赖本地 OHOS 工具链与签名

### 4.3 环境要求

当前代码只区分三类构建环境：

- `development`
- `staging`
- `production`

其中：

- 开发和测试环境允许调整 `baseUrl`
- 正式环境默认隐藏高风险设置项
- 当前仓库默认 `baseUrl` 已切到部署中的 `http://101.35.79.76:8082`，避免应用启动后误连本机 `127.0.0.1:8082`

## 5. 页面与功能明细

### 5.1 启动页 `SplashPage`

功能目标：

- 展示项目名称和品牌感
- 完成基础初始化
- 自动跳转主入口

当前实现：

- 并行初始化本地设置、应用版本信息和登录会话
- 显示加载状态
- 初始化失败时显示错误信息和重试按钮
- 初始化成功后根据登录态跳转主控台或登录页

### 5.2 登录页 `LoginPage`

功能目标：

- 承接账号密码登录和注册
- 展示当前登录模式
- 作为受保护页面前的统一入口

当前实现：

- 支持登录 / 注册模式切换
- 支持账号、密码和确认密码输入
- 认证页壳层已重构为统一软件级背景、响应式头部和双栏工作台
- 当前真实后端已切到 Spring Boot `HttpSession + Cookie` 模式
- 真实接口模式已接通 `/api/register`，注册成功后返回登录模式继续认证
- 支持开发 / 测试环境的联调登录兜底
- 登录成功后直达设备主控台
- 未授权或登录失败时展示明确错误信息

后续待完成：

- 真机验证各平台的 Cookie 会话恢复行为
- 继续收口 Web 跨域部署时的会话策略

### 5.3 首页 `HomePage`

功能目标：

- 作为整个应用的主入口
- 表达当前技术基线和下一阶段规划
- 提供统一导航

当前实现：

- 展示统一品牌总览、版本、当前用户和运行基线说明
- 展示当前后端已接通的登录、视频流接入、设备监控、LED 控制与健康检查能力
- 展示设备摘要快照，包括设备名称、异常等级和主要环境指标
- 提供“视频中心 / 监控主控台 / 系统总览 / 运维设置”四个真实入口
- 首页、主控台、系统总览页与设置页已统一接入工作台壳层，在桌面端使用侧边导航，在紧凑宽度下切换为底部导航
- 不再展示任何未接入后端支撑的识别入口
- 首页结构已拆到 `widgets/` 目录，并通过共享背景层和入口卡片统一视觉语言

### 5.4 视频中心 `VideoHubPage`

功能目标：

- 承接 Java 视频服务下发的视频流清单
- 统一展示流状态、AI 转发状态与播放入口
- 避免在客户端页面层直接拼接媒体地址或读取边缘 AI JSON

当前实现：

- 页面接入统一工作台壳层，并作为一级导航目的地开放
- 调用 `GET /api/video/streams` 拉取视频流列表
- 使用原生卡片展示 `streamId / deviceId / playerUrl / gatewayPageUrl / preferredMode / fallbackMode / aiResultForwarded`
- 支持按流状态进行筛选，并按流标识、设备标识和名称进行检索
- 支持进入单路详情页
- 支持整卡点击进入详情，减少桌面端精确点按成本
- 支持打开外部播放页；当前平台不支持直接拉起时会自动回退为复制链接
- 支持下拉刷新和复制视频服务接口地址
- 当 `19081` 视频服务未返回合法 JSON 时，页面展示正式失败态和重试入口，不使用假数据兜底

后续待完成：

- Java 视频服务稳定返回流清单后的线上实流回归
- 移动端和 OpenHarmony / 鸿蒙平台的系统浏览器拉起适配增强

### 5.5 实时监控页 `RealtimeDetectPage`

功能目标：

- 对齐后端设备监控主控台
- 承接实时状态轮询、错误码映射和 LED 控制
- 为未来摄像头实时识别保留独立扩展空间

当前实现：

- 页面进入后自动启动 `/api/status` 轮询
- 页面包含统一工作台页头、顶部链路状态、设备主横幅、四项环境指标、运行明细、LED 控制和错误码说明区
- 主导航闭环由工作台壳层统一承接，顶部状态区专注轮询状态、刷新控制和退出动作
- 支持手动刷新和自动刷新开关
- LED 控制成功后会继续刷新等待状态回写
- 当前工作区没有发现对应的本地摄像头实时识别后端源码

后续待完成：

- 独立识别服务并入后，为摄像头实时识别恢复独立入口
- 摄像头权限检查
- 预览画面
- 摄像头帧采样与真实实时识别请求
- 真实 overlay 渲染

### 5.6 结果页 `ResultPage`

功能目标：

- 展示一次识别任务的完整结果
- 展示摘要、检测列表、严重程度和防治建议
- 支持把结果保存为历史记录

当前状态：

- 结果页代码和模型仍保留在仓库中
- 当前运行时路由已移除，不作为当前交付能力开放
- 待独立识别服务并入后，再恢复为正式页面

### 5.7 历史记录页 `HistoryPage`

功能目标：

- 浏览本地历史记录
- 查看详情
- 删除单条或清空全部

当前状态：

- 历史记录代码和本地持久化结构仍保留在仓库中
- 当前运行时路由已移除，不作为当前交付能力开放
- 设置页仅保留本机历史清理动作，避免旧数据长期残留

### 5.8 设置页 `SettingsPage`

功能目标：

- 展示环境、平台、版本和服务配置
- 展示当前服务健康状态
- 管理本地数据

当前实现：

- 页面接入统一工作台壳层，与首页、主控台和系统总览保持一致的导航结构
- 展示环境类型、当前平台、应用版本
- 展示当前账号、登录模式和会话到期时间
- 展示 `baseUrl`、连接超时、接收超时
- 展示服务状态、模型状态、服务版本和服务时间
- 支持手动刷新服务健康检查结果
- 仅在开发 / 测试环境开放 `baseUrl` 修改
- 支持清除记住的账号
- 支持清理本机历史记录
- 支持恢复默认设置
- 支持退出登录
- 支持进入系统总览页

当前不提供：

- 通用“清除缓存”能力
- 正式环境下的高风险配置编辑入口

### 5.9 系统总览页 `AboutPage`

功能目标：

- 展示平台定位、业务闭环、栽培背景和调控目标

当前实现：

- 页面接入统一工作台壳层，并通过页头动作统一承接“平台首页 / 主控台 / 登录”跳转
- 顶部 Hero 区聚焦业务价值和运营闭环，而不是静态说明稿
- 页面内容拆分为业务架构、栽培背景、调控目标等独立区块组件
- 主内容区保留响应式断点，不再依赖大文件硬编码布局

## 6. 用户主流程

### 6.1 当前已打通主流程

1. 用户进入启动页
2. 应用初始化本地设置、版本信息和登录会话
3. 未登录用户进入登录页
4. 用户登录成功后进入设备主控台
5. 用户通过顶部导航进入平台总览或运维设置
6. 用户在平台总览查看当前后端能力、设备快照和系统总览入口
7. 用户返回设备主控台继续监控或下发 LED 控制命令

### 6.2 当前已打通辅助流程

1. 用户从登录页进入首页
2. 用户从首页进入设置页
3. 查看环境、平台、版本和当前账号信息
4. 在开发 / 测试环境中修改 `baseUrl`
5. 清除记住账号、清理本机历史记录或恢复默认设置
6. 退出登录或进入系统总览页查看业务说明

### 6.3 当前保留的扩展流程

1. 单图识别与识别历史入口当前保留为扩展位
2. `DetectPage` 默认展示“未接入独立识别服务”说明态
3. 独立识别服务并入后，可恢复结果页和历史记录主链路

### 6.4 当前已打通的设备监控流程

1. 用户从首页进入实时监控页
2. 页面自动轮询 `/api/status`
3. 页面展示设备名称、错误码等级、环境指标和最近刷新时间
4. 用户可手动刷新、暂停自动刷新或下发 LED 控制命令

## 7. 替身数据与真实接口策略

当前策略如下：

- 登录态默认走真实 Spring Boot 接口；开发 / 测试环境仅在显式传入 `USE_MOCK_AUTH=true` 时切到联调登录
- 单图识别链路继续保留真实仓储扩展位，但当前工作区缺少对应识别后端源码，因此默认使用受控 mock
- 历史记录使用本地真实持久化，不使用假列表替身
- 设置页使用本地真实持久化，并对接真实后端设备状态 / LED 控制 / 健康检查
- 实时监控页直接对接真实设备状态后端；摄像头实时识别暂不伪造替身完成态

切换原则：

- 通过 `AuthRepository` / `DetectRepository` / `RealtimeDetectRepository` Provider 切换具体实现，不改页面契约
- 正式环境不允许继续保留假结果分支

## 8. 当前版本验收标准

当前版本至少应满足以下标准：

- `Splash -> Login -> Realtime / Home / Settings / About` 路由可正常使用
- 登录页可完成联调登录、错误提示和未授权回跳
- 首页可展示当前后端能力摘要和设备快照
- 实时监控页可轮询设备状态、展示异常等级并控制 LED
- 设置页可展示环境 / 平台 / 版本信息，并正确控制高风险配置入口与本机数据动作
- `macOS` 与 `OpenHarmony / 鸿蒙` 已纳入平台枚举和工程结构
- 已下线的 `Detect / Result / History` 旧链接会被重定向，不再暴露为可用能力
- 文档说明必须明确区分“已实现”和“待接入”

## 9. 下一阶段重点

下一阶段优先补齐以下能力：

1. 补充独立识别服务后端源码位置或仓库
2. 完成 `Detect` 页面与独立识别服务的正式联调
3. 继续验证跨平台 Cookie 会话恢复
4. 补齐更完整的测试覆盖和多平台构建验证

## 10. 结论

当前功能设计应以“可运行前端基线”作为基准，而不是继续按“纯规划稿”描述。后续每次代码变更后，都必须同步更新本文档中对应页面和流程的实现状态。
