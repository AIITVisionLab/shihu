# 石斛病虫害识别项目 Flutter 前端功能文档

## 1. 文档目标

本文档用于定义石斛病虫害识别项目 Flutter 前端的功能范围、页面职责、交互流程和验收要求，作为后续开发、测试、联调、发布和运维的功能依据。

本文档重点回答以下问题：

- 这个前端要实现哪些功能
- 每个页面承担什么职责
- 用户在各个平台上如何完成主要操作
- 系统应当提供哪些反馈、状态和异常处理
- 在不降低生产标准的前提下，开发与测试环境如何使用受控替身数据

## 2. 产品概述

石斛病虫害识别项目是一个面向石斛种植与养护场景的跨平台前端应用，支持用户通过上传图片或摄像头实时预览的方式识别石斛病害、虫害与健康状态，并查看识别结果、历史记录和基础配置。

产品目标：

- 提供统一、中文、可长期维护的跨平台界面
- 支持 Android、iOS、Web、Windows、Linux 全平台使用
- 满足真实业务使用、测试联调和正式发布要求
- 可平滑对接 Java 后端识别服务并进入正式环境运行

## 3. 目标用户与使用场景

### 3.1 目标用户

- 业务使用者：上传石斛叶片、茎秆或植株图片，查看识别结果和历史记录
- 管理与支持人员：查看版本、平台状态和基础配置
- 开发与测试人员：联调接口、切换测试环境、验证接口返回和异常处理

### 3.2 主要使用场景

- 使用本地石斛图片进行病虫害识别
- 使用摄像头进行实时识别预览
- 查看历史识别记录和结果详情
- 对接本地、测试或局域网 Java 后端进行联调
- 在正式环境中完成稳定运行、异常处理和版本发布

### 3.3 业务对象范围

本项目当前关注的主要识别对象包括：

- 石斛叶片病斑
- 石斛茎秆异常
- 石斛虫害痕迹
- 石斛植株整体健康状态

## 4. 功能范围总览

本项目当前功能范围包括：

- 启动页
- 首页
- 单图识别
- 实时识别
- 结果展示
- 历史记录
- 设置
- 关于项目

全局支持能力包括：

- 全平台统一导航与交互逻辑
- 中文文案
- Material Design 3 风格界面
- 流畅动画与状态反馈
- 开发 / 测试替身数据源与真实接口模式切换
- JSON / HTTP / multipart 与 Java 后端联调预留

### 4.1 历史记录来源策略

当前版本约定如下：

- V1 首版必须支持本地历史记录
- 如果 Java 后端提供历史记录接口，可在 Repository 层切换到远程历史记录实现
- 页面层不感知历史记录来自本地还是远程

## 5. 全局功能要求

### 5.1 平台支持

目标平台：

- Android
- iOS
- Web
- Windows
- Linux

要求：

- 主流程在各平台保持一致
- 页面结构和业务逻辑尽量复用
- 只在权限、窗口尺寸、输入媒介等方面做必要适配

### 5.2 统一交互要求

- 首页入口顺序固定
- 页面返回逻辑统一
- 同类状态使用同类反馈方式
- 按钮、卡片、提示、弹窗使用统一组件风格
- 用户完成一次操作后，系统应立即给出反馈

### 5.3 动画与体验要求

- 页面切换使用统一转场动画
- 重点模块支持轻量入场动画
- 点击、加载、成功、失败必须有反馈
- 动画不影响识别页可读性和交互响应
- 桌面端与 Web 端支持 hover、focus、键盘操作等补充体验

### 5.4 网络与配置要求

- 统一使用 `HTTP / HTTPS`
- 接口数据以 `JSON` 为主
- 图片上传采用 `multipart/form-data`
- 服务地址、超时时间、日志开关集中配置
- 支持后续对接 Java 后端 REST API

### 5.5 安全与质量要求

- 正式环境不允许使用 mock 数据源
- 正式环境不允许通过静默降级核心功能来掩盖真实问题
- 敏感配置和令牌不得明文硬编码
- 关键功能必须具备错误提示、恢复路径和日志记录
- 核心业务流程应具备单元测试、Widget 测试或集成测试覆盖
- 版本发布前应通过静态检查、测试和构建验证

## 6. 页面与功能明细

### 6.1 启动页（SplashPage）

#### 功能目标

- 展示项目 Logo、名称和启动品牌感
- 完成应用初始化
- 自动跳转首页

#### 生产功能要求

- 显示 Logo / 项目名称 / 简短标语
- 延时 1 到 2 秒后进入首页
- 加载环境配置、基础本地状态和必要初始化项

#### 后续可扩展功能

- 读取本地配置
- 检查服务地址
- 检查权限状态
- 检查缓存或历史记录
- 初始化日志、崩溃采集与远程配置

#### 交互要求

- 启动页动画轻量、简洁
- 不允许停留过久
- 初始化失败时给出重试或继续进入选项

### 6.2 首页（HomePage）

#### 功能目标

- 作为整个应用的主入口
- 展示项目定位和主要功能
- 提供统一导航

#### 页面内容

- 项目标题
- 项目简介
- 功能入口卡片或按钮
- 版本信息或状态信息

#### 核心功能

- 进入单图识别页
- 进入实时识别页
- 进入历史记录页
- 进入设置页

#### 交互要求

- 首页视觉完成度要高，适合正式交付和长期使用
- 首屏信息清晰，避免空白感
- 入口按钮明显，主次分明
- 支持轻量入场动画

### 6.3 单图识别页（DetectPage）

#### 功能目标

- 支持用户从相册选择图片或使用相机拍照
- 触发识别流程
- 将结果传递到结果页展示

#### 生产功能要求

- 选择图片
- 预览图片
- 点击开始识别
- 调用真实识别流程并正确处理状态切换

#### 接口与扩展要求

- 上传图片到后端
- 接收识别结果 JSON
- 支持加载态、失败态、重试
- 开发 / 测试构建可切换到受控替身数据源

#### 交互流程

1. 用户进入单图识别页
2. 选择图片或拍照
3. 页面展示所选图片
4. 用户点击开始识别
5. 显示识别中状态
6. 正式环境调用真实接口，开发 / 测试环境可切换替身数据源
7. 成功后跳转结果页
8. 失败时提示错误并支持重试

#### 状态要求

- 未选择图片
- 已选择图片
- 识别中
- 识别成功
- 识别失败

#### 异常处理

- 用户未选择图片时禁止识别
- 读取图片失败时提示重新选择
- 网络异常时提示检查服务地址或网络连接

### 6.4 实时识别页（RealtimeDetectPage）

#### 功能目标

- 打开摄像头进行实时预览
- 展示实时识别观感
- 为后续实时识别能力预留入口

#### 生产功能要求

- 打开摄像头预览
- 显示 overlay 和当前识别状态信息
- 对权限、初始化失败和平台差异进行明确失败处理与提示

#### 接口与扩展要求

- 定时采样图像帧
- 上传帧数据或调用实时识别接口
- 返回框信息并更新 overlay
- 开发 / 测试构建可使用受控替身框数据验证刷新逻辑

#### 页面结构

- 底层：摄像头预览
- 中层：检测框 Overlay
- 顶层：状态栏、识别信息卡片、操作按钮

#### 交互要求

- 识别框更新平滑
- 顶层控件不遮挡主要画面
- 页面响应要稳，避免闪烁和明显掉帧
- 当摄像头不可用时提供说明和退出方式

#### 异常处理

- 摄像头权限被拒绝
- 摄像头初始化失败
- 当前平台不支持相关能力

### 6.5 结果页（ResultPage）

#### 功能目标

- 展示识别结果详情

## 7. 功能流程说明

### 7.1 单图识别流程

1. 用户从首页进入单图识别页
2. 选择图片或拍照
3. 页面展示图片预览
4. 用户点击开始识别
5. 前端进入加载态
6. 正式环境调用真实接口，开发 / 测试环境可切换到受控替身数据源
7. 跳转到结果页展示详情
8. 用户可保存到历史记录

### 7.2 实时识别流程

1. 用户从首页进入实时识别页
2. 应用初始化摄像头
3. 页面展示摄像头预览
4. 正式环境显示真实识别信息，开发 / 测试环境可显示受控 overlay 数据
5. 按固定频率向后端请求识别结果或建立实时通信链路
6. 前端实时更新检测框和结果卡片

### 7.3 历史记录查看流程

1. 用户从首页进入历史记录页
2. 页面加载本地记录
3. 用户查看列表
4. 点击任意记录进入结果详情页

### 7.4 设置修改流程

1. 用户进入设置页
2. 修改服务地址或其他配置
3. 页面执行保存
4. 保存成功后给出提示
5. 后续网络请求使用新配置

## 8. 数据与后端联调需求

### 8.1 基础通信方式

- 协议：`HTTP / HTTPS`
- 数据格式：`JSON`
- 编码：`UTF-8`
- 上传：`multipart/form-data`

### 8.2 推荐接口形式

- `POST /detect/image`：上传图片并返回识别结果
- `GET /history/list`：获取历史记录列表
- `GET /history/{id}`：获取历史详情

### 8.3 推荐返回结构

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

### 8.4 联调要求

- 前端统一由 `ApiClient` 发起请求
- 页面不直接拼接接口地址
- 开发 / 测试环境可在设置页修改 `baseUrl`
- 正式环境通过构建配置、环境变量或远程配置注入 `baseUrl`
- 支持统一处理 401、403、404、500、超时、无网络等异常
- Web 联调时需考虑 Java 后端 CORS 配置
- 请求链路建议附带请求 ID、版本号和设备信息以便排查问题
- 正式环境不通过假数据、空结果或默认成功状态掩盖联调问题

## 9. 开发与测试替身数据要求

- 不进入正式发布环境

本功能文档定义了石斛病虫害识别项目 Flutter 前端当前阶段应具备的核心能力，重点是：

- 全平台统一功能体验
- 单图识别与实时识别两条主流程
- 结果展示、历史记录、设置配置等基础模块
- 以真实接口为基线，开发 / 测试环境可使用受控替身数据
- 视觉完成度、动画流畅度、稳定性和可发布性并重

后续开发应以本文档为功能依据，以架构文档为实现依据，逐步完成从开发版、测试版到正式发布版的迭代。
