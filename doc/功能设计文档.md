# 斛生 客户端功能设计文档

## 1. 文档目标

本文档用于定义斛生客户端当前版本的真实功能范围、页面职责、主流程和验收边界。本文档只以当前仓库代码和 `../shihu-web` 的 `origin/web` 后端源码为准，不再沿用旧规划稿中的预留能力描述。

## 2. 当前版本定位

当前版本是一个可直接用于值守与排障的跨平台客户端基线，重点是把以下能力稳定收进统一工作台：

- 账号登录、注册、会话恢复和退出
- 设备状态轮询与异常等级展示
- LED 补光控制
- 视频中心与视频入口拉起
- 服务健康检查与运行环境管理
- 使用帮助与页面信息说明

当前版本不再提供以下能力：

- 单图识别
- 识别结果页
- 识别历史记录

以上模块已经从代码库和运行时入口中删除，不再作为“预留功能”保留在当前交付基线里。

## 3. 平台范围

目标平台：

- Android
- iOS
- macOS
- Web
- Windows
- Linux
- OpenHarmony / 鸿蒙

平台要求：

- 主流程和页面信息架构在所有平台保持一致
- 首页、主控台、设置页、系统总览页统一使用工作台壳层
- 桌面端使用侧边导航与更宽内容区
- 紧凑宽度下自动切换为底部导航和单列布局
- 全局品牌统一使用透明兰花图标，界面辅助装饰围绕品牌主色展开
- 主视觉统一使用低饱和青松绿主色，辅以雾青、亚麻和淡紫灰点缀，底色保持浅纸感暖灰
- 用户界面不展示独立的配色说明区或品牌信息卡
- 动效、间距、卡片层级和交互反馈在各平台保持统一风格

## 4. 当前功能范围

### 4.1 已交付功能

- 启动初始化 `SplashPage`
- 登录 / 注册 `LoginPage`
- 工作台首页 `HomePage`
- 实时监控主控台 `RealtimeDetectPage`
- 运维设置 `SettingsPage`
- 视频中心 `VideoPage`
- 使用帮助 `AboutPage`
- 本地设置、记住账号、登录会话持久化

### 4.2 当前明确不在范围内的能力

- AI 识别接口接入
- 本地或远程历史记录
- 摄像头权限、取帧和叠框渲染

说明：

- 视频中心已按两份英文命名协作文档接入，但界面层不向普通用户暴露技术接入过程。
- 当前帮助页只保留用户视角的页面说明和查看重点。

## 5. 页面与职责

### 5.1 启动页 `SplashPage`

功能目标：

- 初始化本地设置、应用版本和登录态
- 避免冷启动白屏
- 按登录态跳转登录页或主控台

当前实现：

- 并行读取本地配置和会话
- 展示启动图标区、初始化中和失败重试三类内容
- 初始化完成后直接进入 `LoginPage` 或 `RealtimeDetectPage`

### 5.2 登录页 `LoginPage`

功能目标：

- 统一承接登录和注册
- 作为普通用户的唯一登录入口
- 作为所有受保护页面的唯一认证入口

当前实现：

- 登录字段固定为 `Username / Password / Confirm password`
- 真实接口对接 `/api/login`、`/api/register`、`/api/check-login`、`/api/logout`
- 认证方式固定为 `HttpSession + JSESSIONID`
- 认证页采用桌面说明区 + 独立表单卡片、移动端单列的一体化登录布局
- 登录说明区明确“默认进入值守台”以及总览 / 视频 / 我的的职责，不再复述首页式流程口号
- 登录页不再展示预览入口或其它调试信息；仅在检测到自定义服务配置时展示恢复默认提示，不直出原始服务地址
- 登录成功后直达 `RealtimeDetectPage`

验收边界：

- 不再展示识别、视频或历史记录入口
- 不再使用 Bearer Token 口径误导用户

### 5.3 工作台首页 `HomePage`

功能目标：

- 作为登录后的总览入口
- 优先展示当前设备状态
- 为值守、视频和我的页面提供统一跳转入口

当前实现：

- 顶部总览区使用信号盘面式 Hero，优先展示当前设备结论、最近同步和补光状态
- 桌面端品牌和当前账号统一放在左侧导航仓，页头只保留页面标题与当前页说明
- 使用“值守台 / 视频中心 / 我的”三条常用轨道承接一级入口，不再把入口拆成并列碎卡片
- 环境速览区集中展示设备摘要和四项核心指标，并强化指标矩阵的视觉层级
- 桌面端轨道入口具备轻量 hover 反馈，移动端统一转为单列纵向列表

### 5.4 实时监控页 `RealtimeDetectPage`

功能目标：

- 作为当前版本的核心业务页面
- 持续同步 `/api/status`
- 承接 LED 补光控制

当前实现：

- 页面进入后自动启动轮询
- Hero 区展示设备主状态、当前结论、最近同步与值守节奏
- 指标区展示温度、湿度、光照、MQ2
- 控制区展示设备名称、当前状态、补光状态、最近同步和 LED 开关
- 处理建议区使用“正常 / 注意 / 严重 / 待确认”这类用户视角标签，不直接暴露错误码
- 桌面端优先使用“指标主列 + 操作说明侧列”的两栏布局
- 四项环境指标统一改为矩阵式卡片，移动端优先双项网格
- 设备摘要与 LED 补光控制统一收在同一张操作卡片中

验收边界：

- 页面必须在无设备状态时保持稳定骨架，不允许白屏
- 自动刷新可关闭
- LED 控制提交中不允许重复触发

### 5.5 我的页 `SettingsPage`

功能目标：

- 管理当前设备、登录账号和本机偏好
- 收纳退出登录、清除记住账号和恢复默认等用户操作

当前实现：

- 当前使用总览卡片统一展示设备状态、当前账号、最近同步、数据状态和补光状态
- 当前使用总览采用“设备主卡 + 信息矩阵”的控制台式布局
- 当前账号卡片仅展示当前账号、登录状态和退出入口
- 对不支持安全持久化登录态的平台，当前账号卡片追加明确提示，说明关闭应用后需要重新登录
- 本地偏好卡片支持清除记住账号和恢复默认
- 使用帮助卡片保持在偏好区同侧，避免把帮助信息拆成独立主流程
- 设置页整体保持“总览 Hero + 账号主列 + 偏好 / 帮助侧列”的工作站式结构

验收边界：

- 默认不向普通用户暴露服务地址或健康检查等后台字段
- 当前不再提供清理历史记录等遗留操作

### 5.6 视频中心 `VideoPage`

功能目标：

- 作为视频访问与在线状态核对的统一入口
- 直接承接后端下发的视频地址

当前实现：

- 调用 `GET /api/video/streams`
- 展示画面是否在线、是否可直接在软件内查看以及是否有备用入口
- Android / iOS / macOS / Web 点击画面入口时优先进入软件内播放页，由应用内嵌 WebView 承接后端返回的播放页面
- 当前暂不支持软件内播放的平台，会在播放页明确提示平台能力范围，不再强制拉起外部浏览器
- 视频流卡片采用监看屏式预览舞台加状态信息区，而不是纯文字列表
- 页面默认不展示地址、主机、端口等技术字段
- 大屏下视频列表可扩展为三列，移动端统一回到单列卡片

验收边界：

- 页面获取失败时必须明确展示失败状态，不伪造成功内容
- 页面为空时不展示空壳播放器
- 媒体链路直接走播放地址，不通过业务后端代理视频字节流

### 5.7 使用帮助页 `AboutPage`

功能目标：

- 用用户视角解释页面怎么使用
- 把主路径和查看重点集中放到一页

当前实现：

- 顶部 Hero 区以导览面板形式汇总四个主入口和查看顺序
- 主路径区按轨道式卡片解释“总览 / 值守 / 视频 / 我的”分别什么时候进、进去后做什么
- 信息区只说明用户会看到哪些内容以及平时该先看什么
- 页面整体收敛为导览型界面，不再保留说明文档式长段落结构

## 6. 当前主流程

### 6.1 认证主流程

`SplashPage -> LoginPage -> RealtimeDetectPage -> HomePage / VideoPage / SettingsPage / AboutPage`

### 6.2 值守主流程

`RealtimeDetectPage -> 查看状态 -> 判断异常等级 -> 切换 LED -> 等待状态回写`

### 6.3 排障主流程

`HomePage / RealtimeDetectPage -> 查看当前状态 -> 需要时进入“我的”处理账号或本机偏好 -> 必要时退出并重新登录`

## 7. 设计与体验要求

- 保持 Material 3 信息架构和组件语义
- 透明兰花品牌图标必须在启动页、登录页和工作台导航中保持统一出现
- 页面背景、卡片、动效和导航风格必须统一为同一套浅纸感松青工作台语言
- 背景底色统一控制在偏冷浅雾白范围，避免松青主色被暖米底色带偏
- 工作台壳层统一使用悬浮式导航仓与标题控制仓，不再混用默认页面容器样式
- 装饰层只允许作为弱化背景存在，不使用大面积独立色块抢占正文视觉重心
- 桌面端卡片布局最多两列，手机端优先单列，避免操作页出现碎片化信息流
- 页面文案必须使用用户视角，不直接暴露内部设计概念和开发协作口径
- 默认省略重复说明和跨页重复信息，优先保证主要操作链路清晰
- 顶层模块切换必须使用工作台导航直达，不叠加页面返回栈
- 一级导航切换必须保持已访问板块状态，不重复播放首屏入场动效
- 动效只保留必要反馈，不做炫技装饰
- 在桌面端与移动端都要保证点击路径清晰、响应流畅
- 退出登录、恢复默认设置和清除本机记住账号必须二次确认
- 不允许再把未接入后端的能力当成“占位入口”展示给用户
- 通用卡片统一使用松绿细高光、克制深色渐变面和一致的标题层级，不允许页面各自堆一套装饰语法

## 8. 后续优先级

按当前版本基线，后续优先级如下：

1. 验证 Android、iOS、Windows、macOS、OpenHarmony / 鸿蒙 的真实构建与会话恢复行为
2. 继续优化主控台和设置页在真机上的滚动、输入和反馈细节
3. 未来如需恢复 AI / 视频协作，必须先补齐对应后端源码，再重新立项，不在当前代码库内做预留实现
