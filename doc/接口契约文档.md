# 石斛病虫害识别项目 Flutter 前后端接口契约文档

## 1. 文档目标

本文档用于定义石斛病虫害识别项目 Flutter 前端与 Java 后端之间的接口契约，作为前后端联调、接口实现、模型定义和自动化测试的统一依据。

本文档重点约束：

- 接口路径
- 请求方式
- 请求参数
- 响应结构
- 错误码
- 字段命名
- 图片上传方式
- 正式环境错误处理原则

## 2. 总体约定

### 2.1 协议与格式

- 协议：`HTTP / HTTPS`
- 编码：`UTF-8`
- 普通接口：`application/json`
- 图片上传接口：`multipart/form-data`
- 字段命名：统一使用 `camelCase`
- 时间字段：统一使用 ISO 8601 字符串，例如 `2026-03-07T15:30:00+08:00`

### 2.2 正式环境原则

- 正式环境不允许返回假数据掩盖错误
- 正式环境不允许通过空对象、默认成功值、伪结果替代真实失败
- 接口失败必须返回明确错误码和错误信息
- 前端必须展示明确失败态，而不是静默降级

### 2.3 公共请求头

建议统一携带以下请求头：

- `Content-Type`
- `Accept: application/json`
- `X-Request-Id`
- `X-App-Version`
- `X-Platform`
- `Authorization: Bearer <token>`，如果后续接入鉴权

### 2.4 响应包裹结构

所有业务接口统一返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

字段说明：

- `code`：业务状态码
- `message`：业务提示信息
- `data`：业务数据主体

### 2.5 HTTP 状态码约定

- `200`：请求成功
- `400`：请求参数错误
- `401`：未授权
- `403`：无权限
- `404`：资源不存在
- `409`：资源冲突
- `500`：服务内部错误
- `503`：服务暂不可用

## 3. 业务错误码约定

建议后端至少保留以下业务错误码：

- `200`：成功
- `40001`：参数缺失或格式错误
- `40002`：图片文件无效
- `40003`：图片大小超限
- `40004`：不支持的图片格式
- `40005`：请求频率过高
- `40101`：登录状态失效
- `40301`：无访问权限
- `40401`：记录不存在
- `40901`：重复记录或重复操作
- `50001`：模型推理失败
- `50002`：数据存储失败
- `50003`：服务内部异常
- `50301`：识别服务暂不可用

## 4. 接口列表

当前建议的核心接口如下：

- `GET /api/v1/health`
- `POST /api/v1/detect/image`
- `POST /api/v1/detect/realtime/frame`

说明：

- 当前 V1 版本要求前端具备本地历史记录能力，设置配置也由前端本地管理
- 如果后端提供历史记录能力，可在 Repository 层切换为远程实现，不改页面契约
- 如果后续需要服务端配置，可新增 `/api/v1/config/*` 接口

### 4.1 可选增强接口

如果 Java 后端需要托管历史记录，可额外提供以下增强接口：

- `GET /api/v1/history`
- `GET /api/v1/history/{historyId}`
- `DELETE /api/v1/history/{historyId}`
- `DELETE /api/v1/history`

## 5. 健康检查接口

### 5.1 `GET /api/v1/health`

用途：

- 检查后端服务和识别服务是否可用
- 用于启动页、设置页和联调阶段健康检查

响应示例：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "status": "up",
    "serviceName": "shihu-detect-service",
    "serviceVersion": "1.0.0",
    "modelStatus": "ready",
    "serverTime": "2026-03-07T15:30:00+08:00"
  }
}
```

## 6. 单图识别接口

## 7. 实时识别接口

### 7.1 `POST /api/v1/detect/realtime/frame`

用途：

- 上传实时识别帧
- 返回当前帧检测结果

请求方式：

- `multipart/form-data`

请求字段：

- `file`：当前帧图片，必填
- `sessionId`：实时识别会话 ID，必填
- `frameIndex`：帧序号，必填
- `capturedAt`：采集时间，可选
- `platform`：平台标识，必填

响应示例：

```json

  "message": "success",
  "data": {
    "sessionId": "rt_20260307_0001",
    "frameIndex": 24,
    "summary": {
      "primaryLabelCode": "insect_scale_insect",
      "primaryLabelName": "介壳虫",
      "category": "insect",
      "confidence": 0.9123,
      "severityLevel": "low",
      "healthStatus": "risk"
    },
    "detections": [
      {
        "detectionId": "rt_box_01",
        "labelCode": "insect_scale_insect",
        "labelName": "介壳虫",
        "category": "insect",
        "confidence": 0.9123,
        "severityLevel": "low",
        "bbox": {
          "x": 0.43,
```

## 8. 历史记录接口（可选增强）

说明：

- 本节接口为后端增强能力
- 当前前端首版必须先实现本地历史记录
- 如果后端已提供历史记录接口，前端可在数据源层切换到远程实现

### 8.1 `GET /api/v1/history`

用途：

- 获取历史记录列表

请求参数：

- `pageNum`：页码，默认 `1`
- `pageSize`：每页条数，默认 `20`
- `labelCode`：标签筛选，可选
- `severityLevel`：严重程度筛选，可选
- `startDate`：开始时间，可选
- `endDate`：结束时间，可选

响应示例：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "records": [
      {
        "historyId": "his_20260307_0001",
        "detectId": "det_20260307_0001",
        "coverUrl": "/static/annotated/det_20260307_0001.jpg",
        "primaryLabelCode": "disease_black_spot",
        "primaryLabelName": "黑斑病",
        "category": "disease",
        "severityLevel": "medium",
        "confidence": 0.9721,
        "capturedAt": "2026-03-07T15:30:00+08:00"
      }
    ],
    "total": 1,
    "pageNum": 1,
    "pageSize": 20
  }
}
```

### 8.2 `GET /api/v1/history/{historyId}`

用途：

- 获取单条历史记录详情

响应示例：

- 结构同单图识别返回的 `data` 主体，可额外补充 `historyId`

### 8.3 `DELETE /api/v1/history/{historyId}`

用途：

- 删除单条历史记录

成功响应示例：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "historyId": "his_20260307_0001"
  }
}
```

### 8.4 `DELETE /api/v1/history`

用途：

- 清空历史记录

请求体示例：

```json
{
  "confirm": true
}
```

## 9. 前端解析要求

前端实现必须满足：

- 统一使用 `ApiClient` 解析响应
- 先校验 `code`，再解析 `data`
- `detections` 为空时视为有效结果，但必须显示明确提示
- `bbox` 坐标统一按 `0 ~ 1` 归一化处理
- 正式环境不允许因为字段缺失而静默返回空结果页
- 无法处理的字段异常必须进入错误态并记录日志

## 10. 版本演进要求

如果后续接口升级，建议遵循：

- 路径版本化，例如 `/api/v1/*`
- 新增字段向后兼容
- 删除字段或修改语义必须升级版本
- 响应结构变更必须同步更新数据字典和前端模型

## 11. 结论

本文档用于保证石斛病虫害识别项目的前后端接口实现一致、字段一致、异常处理一致。后续所有 Flutter 网络模型、Java DTO、接口联调和测试样例，均以本文档为准。
