# 石斛病虫害识别项目 Flutter 前后端接口契约文档

## 1. 文档目标

本文档用于定义石斛病虫害识别项目 Flutter 前端与 Java 后端之间的接口契约，作为前后端联调、模型定义和错误处理的统一依据。

说明：

- 接口契约以正式联调目标为准
- 即使当前客户端尚未全部接入真实接口，后续实现也必须遵循本文档

## 2. 总体约定

### 2.1 协议与格式

- 协议：`HTTP / HTTPS`
- 编码：`UTF-8`
- 普通接口：`application/json`
- 图片上传接口：`multipart/form-data`
- 字段命名：统一使用 `camelCase`
- 时间字段：统一使用 ISO 8601 字符串，例如 `2026-03-07T15:30:00+08:00`

### 2.2 正式环境原则

- 正式环境不允许返回假数据掩盖错误
- 正式环境不允许通过空对象、默认成功值或伪结果替代真实失败
- 接口失败必须返回明确错误码和错误信息
- 前端必须展示明确失败态，而不是静默降级

### 2.3 公共请求头

建议统一携带以下请求头：

- `Content-Type`
- `Accept: application/json`
- `X-Request-Id`
- `X-App-Version`
- `X-Platform`
- `Authorization: Bearer <token>`，如果后续接入鉴权

其中 `X-Platform` 和请求体中的 `platform` 建议统一使用以下标准值：

- `android`
- `ios`
- `macos`
- `web`
- `windows`
- `linux`
- `ohos`

兼容说明：

- 当前前端解析层接受 `openharmony`、`harmonyos`、`ohos` 三种输入
- 对外统一值仍应收敛为 `ohos`

### 2.4 响应包裹结构

所有业务接口统一返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

字段说明：

- `code`：业务状态码
- `message`：业务提示信息
- `data`：业务数据主体

### 2.5 HTTP 状态码约定

- `200`：请求成功
- `400`：请求参数错误
- `401`：未授权
- `403`：无权限
- `404`：资源不存在
- `409`：资源冲突
- `500`：服务内部错误
- `503`：服务暂不可用

### 2.6 当前客户端基线说明

当前前端代码状态如下：

- `/api/v1/detect/image` 尚未接入真实调用，页面仍通过受控 mock 仓储返回结果
- `/api/v1/detect/realtime/frame` 仍未接入
- 历史记录与设置当前均为本地管理
- 当前网络层已具备 `ApiClient` 和 `ApiException`，但通用 `ApiResponse<T>` 类型封装仍待补齐

## 3. 业务错误码约定

建议后端至少保留以下业务错误码：

- `200`：成功
- `40001`：参数缺失或格式错误
- `40002`：图片文件无效
- `40003`：图片大小超限
- `40004`：不支持的图片格式
- `40005`：请求频率过高
- `40101`：登录状态失效
- `40301`：无访问权限
- `40401`：记录不存在
- `40901`：重复记录或重复操作
- `50001`：模型推理失败
- `50002`：数据存储失败
- `50003`：服务内部异常
- `50301`：识别服务暂不可用

## 4. 核心接口列表

当前核心接口如下：

- `GET /api/v1/health`
- `POST /api/v1/detect/image`
- `POST /api/v1/detect/realtime/frame`

说明：

- 当前 V1 版本要求前端具备本地历史记录能力，设置配置也由前端本地管理
- 如果后端后续提供历史记录能力，可在 Repository 层切换为远程实现，不改页面契约

## 5. 健康检查接口

### 5.1 `GET /api/v1/health`

用途：

- 检查后端服务和识别服务是否可用

## 6. 单图识别接口

### 6.1 `POST /api/v1/detect/image`

- 上传一张石斛图片

- `multipart/form-data`

请求字段：

- `file`：图片文件，必填
- `clientTraceId`：客户端请求标识，必填
- `capturedAt`：采集时间，可选
- `platform`：平台标识，必填
- `imageWidth`：图片宽度，可选
- `imageHeight`：图片高度，可选

`platform` 取值：

- `android / ios / macos / web / windows / linux / ohos`

## 7. 实时识别接口

### 7.1 `POST /api/v1/detect/realtime/frame`

用途：

- 上传实时识别帧
- 返回当前帧检测结果

请求方式：

- `multipart/form-data`

请求字段：

- `file`：当前帧图片，必填
- `sessionId`：实时识别会话 ID，必填
- `frameIndex`：帧序号，必填
- `capturedAt`：采集时间，可选
- `platform`：平台标识，必填

响应结构：

- 与单图识别接口的 `data` 字段结构保持一致
- `sourceType` 固定为 `realtime`

说明：

- 当前前端尚未接入该接口
- 接入后不得通过假预览或假框伪装成真实识别

## 8. 可选增强接口

如果后端需要托管历史记录，可额外提供以下接口：

- `GET /api/v1/history`
- `GET /api/v1/history/{historyId}`
- `DELETE /api/v1/history/{historyId}`
- `DELETE /api/v1/history`

说明：

- 当前页面层只依赖 `HistoryRecord` 契约
- 本地与远程切换应收敛在 Repository 层

## 9. 契约执行要求

后续联调时必须满足：

- 接口字段与本文档一致
- 平台值与 `PlatformType` 枚举一致
- 前端和后端对错误码语义保持一致
- 文档、模型、请求实现同步更新

## 10. 结论

接口契约文档现在既服务于未来联调，也用于约束当前前端继续补齐真实接口实现。任何接口路径、字段、枚举或错误码的变动，都必须同步更新本文档和对应模型。
