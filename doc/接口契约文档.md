# 石斛项目 Flutter 前后端接口契约文档

## 1. 文档目标

本文档不再以旧版规划稿为准，而是以当前工作区内实际可见的后端源码为准，说明 Flutter 前端与后端的真实接口契约、会话方式和当前联调边界。

当前已确认的本地后端工程：

- 同级目录 `../shihu-web`
- 技术栈：Spring Boot
- 真实控制器：`LoginController`、`DeviceApiController`

## 2. 当前已确认的后端接口

当前工作区里可以确认的后端接口只有以下 6 个：

- `POST /api/login`
- `GET /api/check-login`
- `POST /api/logout`
- `GET /api/status`
- `POST /api/ops/led`
- `GET /api/health`

说明：

- 上述接口均来自 `../shihu-web` 的真实控制器源码。
- 当前工作区里没有发现 `/api/v1/auth/*`、`/api/v1/detect/image`、`/api/v1/detect/realtime/frame` 对应的后端实现。
- 因此前端文档和页面中凡是把这些 `/api/v1/*` 路径描述成“本地已联调完成”的内容，均视为过时描述。

## 3. 认证契约

### 3.1 会话模式

当前后端认证不是 Bearer Token 模式，而是：

- `HttpSession`
- `JSESSIONID` Cookie

前端约束：

- Web 端通过 `withCredentials` 交由浏览器自动管理 Cookie
- 非 Web 端需要保存并回传会话 Cookie
- 不再把旧文档中的 `refreshToken`、`Authorization: Bearer <token>` 当成当前后端真实要求

### 3.2 `POST /api/login`

用途：

- 账号密码登录
- 登录成功后由后端在响应头写入会话 Cookie

请求体：

```json
{
  "username": "demo",
  "password": "demo123456"
}
```

响应体：

```json
{
  "success": true,
  "message": "登录成功"
}
```

失败响应示例：

```json
{
  "success": false,
  "message": "用户名或密码错误"
}
```

字段说明：

- `success`：`bool`，是否登录成功
- `message`：`String`，登录结果说明

### 3.3 `GET /api/check-login`

用途：

- 检查当前会话是否仍然有效
- 启动恢复登录态时由前端主动调用

成功响应示例：

```json
{
  "loggedIn": true,
  "username": "demo"
}
```

未登录响应示例：

```json
{
  "loggedIn": false
}
```

字段说明：

- `loggedIn`：`bool`，当前会话是否有效
- `username`：`String?`，登录成功时返回

### 3.4 `POST /api/logout`

用途：

- 注销当前会话

请求体：

- 无强制字段

响应体：

```json
{
  "success": true,
  "message": "已登出"
}
```

## 4. 设备状态接口

### 4.1 `GET /api/status`

用途：

- 拉取当前设备缓存状态
- 用于设备监测、设置页状态卡片和后续实时监控页面

响应结构：

```json
{
  "deviceId": "dev_001",
  "deviceName": "石斛培育柜",
  "temperature": { "value": 24.5, "unit": "°C" },
  "humidity": { "value": 82.0, "unit": "%" },
  "light": { "value": 1500.0, "unit": "Lux" },
  "mq2": { "value": 18.0, "unit": "ppm" },
  "error": { "value": 0 },
  "led": { "value": true },
  "updatedAt": 1741399200000
}
```

字段说明：

- `deviceId`：`String`
- `deviceName`：`String`
- `temperature / humidity / light / mq2`：对象结构，包含 `value` 与 `unit`
- `error.value`：`int`，当前错误码
- `led.value`：`bool`，LED 当前状态
- `updatedAt`：`long`，毫秒时间戳

错误码语义：

- `0`：系统运行正常
- `1`：系统预警，建议人工复核
- `2`：系统严重告警
- 其他值：前端按未知状态展示

## 5. LED 控制接口

### 5.1 `POST /api/ops/led`

用途：

- 控制设备 LED 开关

请求体：

```json
{
  "deviceId": "dev_001",
  "deviceName": "石斛培育柜",
  "led": true
}
```

响应体：

```json
{
  "status": "accepted",
  "requestId": "req_001",
  "message": "指令已下发"
}
```

说明：

- 控制器返回 HTTP `202 Accepted`
- 前端当前把 `accepted / success / ok` 都视为成功

## 6. 健康检查接口

### 6.1 `GET /api/health`

用途：

- 快速检查后端服务是否存活

当前后端返回：

```text
ok
```

前端处理约定：

- 返回字符串 `ok` 视为服务正常
- 当前前端会将其映射为 `ServiceHealthInfo(status: "up", serviceName: "iot-onenet", serviceVersion: "web", modelStatus: "ready")`

## 7. 错误处理约定

当前本地后端并没有统一的 `ApiResponse<T>` 包裹结构，至少以下接口不是：

- `/api/login`
- `/api/check-login`
- `/api/logout`
- `/api/status`
- `/api/ops/led`
- `/api/health`

补充说明：

- 登录校验失败返回 `{ success: false, message }`
- 参数校验异常由 `GlobalExceptionHandler` 返回 `400`
- LED 控制异常通常返回 `{ status: "error", requestId: null, message }`
- 因此前端必须按接口类型分别解析，不能再强行假设全部接口统一包裹 `code / message / data`

## 8. 前端当前实现约束

当前 Flutter 前端已经按以下原则收口：

- `AuthRepository` 真实实现对接 `/api/login`、`/api/check-login`、`/api/logout`
- `ApiClient` 在 Web 端启用 `withCredentials`
- `ApiClient` 在非 Web 端支持附带持久化的会话 Cookie
- `ServiceHealthRepository` 只对接 `/api/health`
- `DeviceStateRepository` 对接 `/api/status` 与 `/api/ops/led`

## 9. 当前未确认的识别接口

以下接口在 Flutter 前端代码里仍有仓储或页面预留，但当前工作区没有发现对应后端源码：

- `/api/v1/detect/image`
- `/api/v1/detect/realtime/frame`

因此当前结论必须写清楚：

- 前端仍保留识别相关模型、页面和仓储扩展点
- 但不能把这些接口写成“当前本地后端已经完成联调”
- 如果后续要继续对接，必须先补充独立识别服务的真实仓库位置或源码
