# 病虫害识别项目 Flutter 前端架构设计文档

## 1. 文档目标

本文档用于指导病虫害识别项目的 Flutter 跨平台前端开发，目标是搭建一套：

- 中文界面
- 符合 Material Design / Google 设计语言
- 结构清晰、模块解耦
- 便于多人协作与后续扩展
- 即使当前无后端，也能先完成前端框架与静态演示

本文档面向当前前端开发负责人，适用于从 0 开始搭建项目骨架。

## 2. 前端建设目标

本项目的前端不是简单“写几个页面”，而是要承担以下职责：

1. 提供跨平台统一的用户界面
2. 承担图片选择、摄像头预览、结果展示等核心交互
3. 与后端识别接口解耦联调
4. 通过清晰的视觉表达提升作品完成度与答辩观感
5. 为后续接入实时识别、历史记录、设置配置、多平台适配预留扩展能力
6. 支持 Android、iOS、Web、Windows、Linux 的统一开发与演示

因此，前端必须优先考虑：

- 结构化
- 可维护
- 可演示
- 可扩展

### 2.1 平台覆盖范围

本项目按全平台方案设计，当前目标平台包括：

- Android
- iOS
- Web
- Windows
- Linux

平台策略：

- 业务代码尽量共用，同一功能不按平台重复实现
- UI 保持统一设计语言，只针对屏幕尺寸和交互媒介做响应式调整
- 优先保证 Android、Windows、Linux 演示效果稳定
- iOS 和 Web 保持同样的信息架构与操作流程

## 3. 设计原则

### 3.1 中文优先

项目面向国内比赛与中文答辩场景，界面文案统一采用中文，避免中英文混杂。

文案示例：

- 开始识别
- 实时监测
- 历史记录
- 检测结果
- 严重程度
- 防治建议
- 设置
- 关于项目

### 3.2 Material Design 风格

UI 采用 Google Material Design 3 风格，整体方向：

- 简洁
- 明确的信息层级
- 卡片式内容组织
- 圆角、留白、阴影适中
- 主色偏绿色 / 农业风

建议视觉关键词：

- 自然绿色
- 浅色背景
- 白色卡片
- 圆角按钮
- 信息块清晰分区

### 3.3 解耦优先

页面、数据、网络、业务逻辑、公共组件必须分层，不允许所有逻辑堆在页面文件中。

### 3.4 可扩展优先

当前即使无后端，也要预留：

- 接口层
- 数据模型层
- 摄像头实时识别页
- 本地历史记录层
- 设置层

这样后续对接时不需要推倒重来。

### 3.5 视觉完成度优先

项目用于比赛展示与答辩，界面不能只满足“能用”，还要具备较高完成度。

视觉目标：

- 首页有明显的作品气质，而不是默认脚手架页面
- 关键页面有统一的色彩、圆角、阴影、间距和排版规则
- 信息展示清晰，同时具备一定设计感
- 适合截图、录屏、答辩展示

设计要求：

- 建立统一的设计令牌，如主色、辅助色、圆角、间距、字号、卡片阴影
- 核心页面保持一致的标题区、内容区、操作区层级
- 避免页面元素随意堆叠，优先使用卡片化和分区式布局
- 首页、结果页、实时识别页必须作为视觉重点页面重点打磨

### 3.6 跨平台交互一致性优先

本项目需要兼顾 Android、iOS、Web、Windows、Linux 展示，因此必须做到“界面可自适应，交互逻辑尽量一致”。

一致性原则：

- 相同功能在不同平台保持相同入口位置和相同操作流程
- 相同状态使用相同反馈方式，如 loading、空状态、错误提示、成功提示
- 相同业务页面使用同一套 controller / service 逻辑，不按平台复制业务代码
- 平台差异只体现在权限申请、窗口尺寸、自适应布局等必要层面，不体现在核心交互路径上

具体要求：

- 首页导航入口顺序固定，不因平台变化而调整主要功能位置
- 单图识别、实时识别、历史查看、设置等主流程在各平台保持一致
- 按钮文案、图标语义、返回逻辑、结果展示结构保持统一
- 使用统一组件库封装按钮、卡片、弹窗、列表项和状态页，减少平台间样式漂移
- 通过响应式布局适配尺寸变化，而不是为不同平台设计不同业务流程
- Windows 与 Linux 端优先复用与移动端相同的页面结构和状态流转

## 4. 页面规划

建议前端至少包含以下页面：

### 4.1 启动页（SplashPage）

作用：

- 启动过渡
- 初始化配置
- 后续可扩展为服务检测、权限检查、缓存读取

当前阶段：

- 显示 Logo / 项目名称
- 自动跳转首页

### 4.2 首页（HomePage）

作用：

- 作为项目主入口
- 展示主要功能
- 提供统一导航

建议内容：

- 项目标题
- 一句简介
- 开始识别
- 实时监测
- 历史记录
- 设置

### 4.3 单图识别页（DetectPage）

作用：

- 选择图片
- 拍照上传
- 触发识别

当前阶段：

- 可先做静态交互
- 无后端时使用 mock 数据模拟识别结果

### 4.4 实时识别页（RealtimeDetectPage）

作用：

- 打开摄像头
- 显示 CameraPreview
- 后续承接实时识别与框叠加

当前阶段：

- 先完成摄像头预览
- 再完成假框 overlay

### 4.5 结果页（ResultPage）

作用：

- 展示识别结果
- 展示目标框、类别、置信度、严重程度、防治建议

这是最重要的业务页面。

### 4.6 历史记录页（HistoryPage）

作用：

- 展示历史检测结果
- 支持点击查看详情

### 4.7 设置页（SettingsPage）

作用：

- 配置服务地址
- 查看平台信息
- 查看版本号
- 清除缓存 / 清除记录

### 4.8 关于页（AboutPage，可选）

作用：

- 展示项目简介、技术栈、团队信息
- 也可合并到设置页中

## 5. 页面流转关系

推荐路由流如下：

- SplashPage -> HomePage
- HomePage -> DetectPage
- HomePage -> RealtimeDetectPage
- HomePage -> HistoryPage
- HomePage -> SettingsPage
- DetectPage -> ResultPage
- HistoryPage -> ResultPage
- SettingsPage -> AboutPage（可选）

## 6. 推荐前端架构

采用：

Feature-first + 分层解耦 + 可替换数据源

即：

- 按功能拆模块
- 公共能力抽离
- 页面不直接依赖底层实现
- mock 数据与真实接口可无缝切换

### 6.1 分层说明

#### App 层

负责全局配置：

- App 入口
- 路由
- 主题

#### Core 层

负责基础设施：

- 网络请求
- 本地存储
- 常量
- 工具类

#### Shared 层

负责公共能力：

- 公共组件
- 通用模型
- 可复用的通用 UI

#### Features 层

按业务功能拆分：

- splash
- home
- detect
- realtime_detect
- result
- history
- settings

## 8. 解耦原则

### 8.1 页面不直接写死接口实现

页面层只调用 service / controller，不直接写网络请求。

错误示例：

- 在页面按钮点击事件里直接写 HTTP 请求

正确做法：

- 页面调用 `detectController.startDetect()`
- controller 调用 `detectService.detectImage()`
- service 再决定返回 mock 还是 API 数据

### 8.2 页面不直接解析 JSON

JSON 解析应放在 model / service 层。

### 8.3 页面不直接管理复杂状态

页面只负责展示，识别中、识别成功、识别失败等状态应通过 controller/provider 管理。

### 8.4 公共组件统一抽离

例如：

- 通用按钮
- 通用卡片
- 空状态页
- 加载组件

不能在每个页面重复拷贝。

## 9. 状态管理建议

优点：

- 结构更现代
- 可测试性更好
- 扩展性更强

建议：

如果当前时间紧，优先用 Provider；如果项目会持续迭代，优先用 Riverpod。就比赛项目的性价比来看，Provider 是更稳妥的首选。

## 10. 无后端阶段的开发策略

由于当前无后端，建议采用：

Mock First 开发模式

即：

1. 先定义数据模型
2. 再写假数据
3. 页面通过 service 获取假数据
4. 后续后端完成后，只替换 service 层数据源

### 10.1 推荐数据流

```text
Page
 -> Controller / Provider
   -> Service
     -> MockDataSource（当前）
     -> ApiDataSource（未来）
```

### 10.2 当前阶段必须完成的 mock 内容

- 检测结果 mock 数据
- 历史记录 mock 数据
- 实时识别假框 mock 数据
- 设置页配置 mock 数据

## 11. 重点页面设计建议

### 11.1 首页设计

建议布局：

- 顶部标题区
- 作品简介区
- 功能入口区
- 底部状态 / 版本信息

首页重点是：

- 不空
- 不像测试页
- 一眼能看懂作品在干什么
- 有明显视觉主次和作品感

### 11.2 结果页设计

建议布局：

- 原图区域 / 结果图区域
- 检测结果信息卡片
- 严重程度卡片
- 防治建议卡片
- 保存记录按钮

结果页必须做到：

- 信息清楚
- 层次明显
- 适合录视频展示
- 图片、结果、建议三个区域分区明确

### 11.3 实时识别页设计

建议使用 `Stack`：

- 底层：摄像头预览
- 中层：检测框 Overlay
- 顶层：状态栏 + 结果卡片

这个页面的重点不是花哨，而是：

- 稳定
- 清楚
- 有“实时识别”的观感
- 控件层不遮挡核心识别画面

## 12. Material Design 风格建议

### 12.1 色彩建议

- 主色：绿色系，如 `Color(0xFF2E7D32)`
- 辅助色：浅绿色 / 青绿色
- 背景色：浅灰白
- 卡片色：纯白
- 错误色：红色
- 警告色：橙色

### 12.2 组件风格建议

- Button：圆角 + 统一高度
- Card：中等圆角 + 轻阴影
- 输入框：简洁边框
- AppBar：简洁，不要堆很多元素
- 图标：统一使用 Material Icons
- 弹窗 / BottomSheet / SnackBar 风格统一

### 12.3 文本层级建议

- 一级标题：页面标题
- 二级标题：模块标题
- 正文：识别说明、建议内容
- 辅助文字：时间、置信度、状态

### 12.4 视觉细节建议

- 首页可加入农业识别相关插画、渐变背景或品牌区块，避免过于朴素
- 卡片之间保持统一留白，避免页面拥挤
- 重点操作按钮使用高对比主色，次级操作降低视觉权重
- 结果页中的严重程度、置信度、建议内容可通过标签色和信息块强化识别效率
- 所有页面遵循统一栅格和边距规则，避免不同页面像不同项目拼接而成

### 12.5 跨平台交互规范

- 导航层级统一：启动页 -> 首页 -> 功能页 -> 结果详情页
- 返回行为统一：详情返回上一级，首页作为主入口不出现歧义返回
- 主要操作入口统一放在页面主要可视区域，不因平台变化而移动到完全不同位置
- 加载、错误、空数据、成功保存等反馈文案与反馈形式统一
- 仅在必须时处理平台差异，例如摄像头权限弹窗、Windows / Linux 窗口缩放、Web 文件选择
- 不在 iOS 单独设计一套路由逻辑，也不在 Web 单独设计另一套结果流程
- 桌面端操作仍以同一套页面流程为主，不额外拆分成独立业务版界面

## 15. 当前阶段最小可交付物

即使没有后端，前端也应该先交付以下内容：

- 可运行的 Flutter 项目
- 完整目录结构
- Material 风格统一主题
- 首页、单图识别页、实时识别页、结果页、历史记录页、设置页
- mock 数据驱动的结果展示
- 摄像头预览
- 假框 overlay 演示
- service 层预留
- 至少可在 Android、Windows、Linux 上稳定运行，其他平台保持同结构可扩展

这套内容已经足以支撑前端框架汇报与后续快速联调。

## 16. 推荐依赖

建议优先选择成熟、社区常用、便于比赛项目快速落地的依赖：

- `provider`：状态管理
- `go_router` 或 `Navigator 2.0` 封装：路由管理
- `dio`：网络请求
- `shared_preferences`：轻量配置存储
- `path_provider`：本地文件路径
- `image_picker`：相册与拍照选图
- `camera`：实时摄像头预览
- `package_info_plus`：版本信息
- `device_info_plus`：设备信息

原则：

- 当前阶段避免一次性引入过多复杂依赖
- 能用官方 / 主流插件解决的问题，不自造轮子
- 依赖引入要服务于可演示和可交付，而不是堆技术名词
- 选择依赖前优先检查 Android、iOS、Web、Windows、Linux 的插件兼容性

## 17. 验收标准建议

为了避免“页面做了很多但不可演示”，建议在文档中明确最小验收标准：

### 17.1 页面验收

- 所有核心页面可正常跳转
- 页面文案统一中文
- 主题、色彩、按钮风格统一
- 空状态、加载态、失败态有基本占位
- 首页、结果页、实时识别页具备可展示的视觉完成度

### 17.2 交互验收

- 单图识别流程可完整演示
- mock 结果可正确传递到结果页
- 历史记录可新增、可查看详情
- 实时识别页可显示预览和假框
- Android、iOS、Web、Windows、Linux 的主流程入口和操作顺序保持一致

### 17.3 架构验收

- 页面层不直接写网络请求
- 数据模型、service、controller 已拆分
- mock 数据源可独立替换
- 路由与主题集中管理
- 公共组件已抽离，避免不同平台各自实现一套 UI 逻辑
- 对桌面端不单独复制业务层代码，平台差异控制在适配层

## 18. 结论

本项目 Flutter 前端应以“中文、Material 风格、模块解耦、可扩展、全平台一致性”为核心目标，采用 Feature-first 分层架构，优先搭建：

- 页面框架
- 路由体系
- 统一主题
- mock 数据流
- 摄像头预览页
- 结果展示页

在无后端阶段，前端仍可独立完成大部分框架与界面建设；后续接入识别服务时，只需替换数据层与 service 层实现，无需大规模重构。
