# 石斛病虫害识别项目 Flutter 前端架构设计文档

## 1. 文档目标

本文档用于指导石斛病虫害识别项目的 Flutter 跨平台前端开发，目标是搭建一套：

- 中文界面
- 符合 Material Design / Google 设计语言
- 结构清晰、模块解耦
- 便于多人协作与后续扩展
- 即使后端暂未完全就绪，也能先完成可联调、可测试、可发布的前端框架

本文档面向当前前端开发负责人，适用于从 0 开始搭建项目骨架。

## 2. 前端建设目标

本项目的前端不是简单“写几个页面”，而是要承担以下职责：

1. 提供跨平台统一的用户界面
2. 承担图片选择、摄像头预览、结果展示等核心交互
3. 与后端识别接口解耦联调
4. 通过清晰的视觉表达提升产品完成度与信息表达质量
5. 为后续接入实时识别、历史记录、设置配置、多平台适配预留扩展能力
6. 支持 Android、iOS、Web、Windows、Linux 的统一开发、测试与发布

项目业务范围：

- 聚焦石斛种植场景
- 覆盖石斛病害、虫害及健康状态识别
- 支持石斛叶片、茎秆、植株图像的采集与结果分析

因此，前端必须优先考虑：

- 结构化
- 可维护
- 可验证
- 可扩展
- 可发布

### 2.1 平台覆盖范围

本项目按全平台方案设计，当前目标平台包括：

- Android
- iOS
- Web
- Windows
- Linux

平台策略：

- 业务代码尽量共用，同一功能不按平台重复实现
- UI 保持统一设计语言，只针对屏幕尺寸和交互媒介做响应式调整
- 优先保证 Android、Windows、Linux 的发布稳定性与运行一致性
- iOS 和 Web 保持同样的信息架构与操作流程

## 3. 设计原则

### 3.1 中文优先

项目主要面向中文业务场景与中文用户，界面文案统一采用中文，避免中英文混杂。

文案示例：

- 开始识别
- 实时监测
- 历史记录
- 检测结果
- 石斛病害类型
- 石斛虫害类型
- 严重程度
- 防治建议
- 设置
- 关于项目

### 3.2 Material Design 风格

UI 采用 Google Material Design 3 风格，整体方向：

- 简洁
- 明确的信息层级
- 卡片式内容组织
- 圆角、留白、阴影适中
- 主色偏绿色 / 科技农业风

建议视觉关键词：

- 自然绿色
- 石斛生态感
- 浅色背景
- 白色卡片
- 圆角按钮
- 信息块清晰分区

### 3.3 解耦优先

页面、数据、网络、业务逻辑、公共组件必须分层，不允许所有逻辑堆在页面文件中。

### 3.4 可扩展优先

当前即使无后端，也要预留：

- 接口层
- 数据模型层
- 摄像头实时识别页
- 本地历史记录层
- 设置层

这样后续对接时不需要推倒重来。

### 3.5 视觉完成度优先

项目需要面向真实使用和长期维护，界面不能只满足“能用”，还要具备较高完成度。

视觉目标：

- 首页有明显的作品气质，而不是默认脚手架页面
- 关键页面有统一的色彩、圆角、阴影、间距和排版规则
- 信息展示清晰，同时具备一定设计感
- 适合日常使用、培训交付和正式汇报

设计要求：

- 建立统一的设计令牌，如主色、辅助色、圆角、间距、字号、卡片阴影
- 核心页面保持一致的标题区、内容区、操作区层级
- 避免页面元素随意堆叠，优先使用卡片化和分区式布局
- 首页、结果页、实时识别页必须作为视觉重点页面重点打磨

### 3.6 跨平台交互一致性优先

本项目需要兼顾 Android、iOS、Web、Windows、Linux 展示，因此必须做到“界面可自适应，交互逻辑尽量一致”。

一致性原则：

- 相同功能在不同平台保持相同入口位置和相同操作流程
- 相同状态使用相同反馈方式，如 loading、空状态、错误提示、成功提示
- 相同业务页面使用同一套 controller / service 逻辑，不按平台复制业务代码
- 平台差异只体现在权限申请、窗口尺寸、自适应布局等必要层面，不体现在核心交互路径上

具体要求：

- 首页导航入口顺序固定，不因平台变化而调整主要功能位置
- 单图识别、实时识别、历史查看、设置等主流程在各平台保持一致
- 按钮文案、图标语义、返回逻辑、结果展示结构保持统一
- 使用统一组件库封装按钮、卡片、弹窗、列表项和状态页，减少平台间样式漂移
- 通过响应式布局适配尺寸变化，而不是为不同平台设计不同业务流程
- Windows 与 Linux 端优先复用与移动端相同的页面结构和状态流转

### 3.7 动效与使用体验优先

项目不仅要“看起来完整”，还要“用起来顺手”。动画的职责是帮助用户理解界面变化、感知状态切换，而不是单纯装饰。

动效原则：

- 所有动画服务于状态变化、页面切换和信息层级表达
- 动画要轻、准、稳，避免拖沓和眩晕感
- 同类组件使用统一的动效节奏、曲线和时长
- 动画在 Android、iOS、Web、Windows、Linux 上保持相近体验

体验原则：

- 用户操作后应尽快得到反馈，不能点击后无响应
- 加载、识别中、保存成功、失败重试等状态必须有明确反馈
- 重点页面优先保证流畅性，再考虑复杂效果
- 桌面端、Web 端要补充 hover、focus、键盘交互等体验细节

## 6. 推荐前端架构

采用：

Feature-first + 分层解耦 + 可替换数据源

即：

- 按功能拆模块
- 公共能力抽离
- 页面不直接依赖底层实现
- 开发替身数据源与真实接口可切换，正式环境固定使用真实接口

### 6.1 分层说明

#### App 层

负责全局配置：

- App 入口
- 路由
- 主题

#### Core 层

负责基础设施：

- 网络请求
- HTTP 配置
- JSON 序列化与反序列化
- 请求拦截与统一错误处理
- 本地存储
- 常量
- 工具类

#### Shared 层

负责公共能力：

- 公共组件
- 通用模型
- 可复用的通用 UI

#### Features 层

按业务功能拆分：

- splash
- home
- detect
- realtime_detect
- result
- history
- settings

### 6.2 接口与 Java 后端对接约定

为了后续顺利对接 Java 后端并满足生产环境要求，前端基础通信方案建议统一为：

- 通信协议：`HTTP / HTTPS`
- 数据格式：`application/json`
- 字符编码：`UTF-8`
- 文件上传：`multipart/form-data`
- 接口风格：优先采用 RESTful 风格

推荐按常见 Java 后端项目方式约定接口：

- Java 后端建议基于 Spring Boot 等常规 REST API 方式提供服务
- Flutter 前端通过 `Dio` 或同类 HTTP 客户端统一管理请求
- 所有接口地址由统一配置管理，不在页面中写死 IP 和端口
- 所有 JSON 字段命名风格统一，建议前后端都使用 `camelCase`

推荐响应结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

推荐约定：

- `code`：业务状态码
- `message`：提示信息
- `data`：实际业务数据

如果后端需要返回分页数据，可扩展为：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "records": [],
    "total": 0,
    "pageNum": 1,
    "pageSize": 10
  }
}
```

图片识别类接口建议：

- 单图识别：前端上传图片文件，后端返回识别结果 JSON
- 历史记录：前端通过 HTTP 获取记录列表和详情
- 设置配置：服务地址、超时时间等在本地保存，不依赖后端
- 实时识别：开发环境可先使用受控替身数据源；生产环境建议采用轮询、HTTP 分帧上传，或 WebSocket

建议统一请求头：

- `Content-Type: application/json`
- `Accept: application/json`
- `Authorization: Bearer <token>`，如果后端后续加入鉴权

建议统一错误处理：

- HTTP 错误码和业务错误码分开处理
- 网络超时、连接失败、服务异常、数据为空分别给出不同提示
- 正式环境不允许通过返回假数据、空结果或伪成功状态来静默绕过错误
- 401、403、500 等常见错误在 `api_client.dart` 中集中处理
- Web 平台联调时额外关注跨域配置，Java 后端需提前处理 CORS

建议统一环境配置：

- 开发环境、测试环境、预发环境、正式环境使用不同 `baseUrl`
- `baseUrl`、超时时间、是否开启日志打印等放入统一配置层
- 设置页仅在开发 / 测试构建中允许手动修改服务地址
- 正式环境通过构建配置、环境变量或远程配置注入服务地址
- 不把接口地址和鉴权信息散落在多个页面文件中

### 6.3 生产环境工程要求

为了达到生产环境标准，前端工程必须额外满足以下要求：

- 环境隔离：开发、测试、预发、正式环境配置严格分离
- 配置管理：敏感配置不写死在代码中，正式环境通过安全配置方式注入
- 安全存储：令牌、设备标识等敏感信息使用安全存储方案管理
- 日志治理：区分调试日志与正式日志，正式环境禁止输出敏感信息
- 可观测性：接入崩溃采集、关键错误上报、关键链路日志与性能指标
- 失败策略：超时、重试、不可用状态和明确失败提示必须统一处理，不允许静默降级核心能力
- 质量门禁：提交前必须通过格式化、静态检查、单元测试和关键集成测试
- 发布流程：构建、签名、版本号、变更记录和回滚策略要有明确规范

## 8. 解耦原则

### 8.1 页面不直接写死接口实现

页面层只调用 service / controller，不直接写网络请求。

错误示例：

- 在页面按钮点击事件里直接写 HTTP 请求

正确做法：

- 页面调用 `detectController.startDetect()`
- controller 调用 `detectService.detectImage()`
- service 在开发 / 测试构建中可切换替身数据源，在正式环境固定调用 API

### 8.2 页面不直接解析 JSON

JSON 解析应放在 model / service 层。

### 8.3 页面不直接管理复杂状态

页面只负责展示，识别中、识别成功、识别失败等状态应通过 controller/provider 管理。

### 8.4 公共组件统一抽离

例如：

- 通用按钮
- 通用卡片
- 空状态页
- 加载组件

不能在每个页面重复拷贝。

## 9. 状态管理建议

生产环境推荐：

### 方案 A：Riverpod

优点：

- 依赖注入与状态管理边界更清晰
- 可测试性更好
- 更适合复杂状态和长期维护
- 更适合多平台生产项目

### 方案 B：Provider

优点：

- 学习成本较低
- 对简单模块上手较快
- 可作为低复杂度场景的补充方案

建议：

如果按生产环境标准建设，优先使用 Riverpod；只有在模块非常简单且团队已有既有沉淀时，才考虑继续使用 Provider。

## 10. 开发、测试与联调阶段策略

即使后端暂未完全就绪，也不应以演示思路组织前端，而应采用：

Contract First + 受控替身数据源

即：

1. 先定义数据模型
2. 再定义接口契约和错误码规范
3. 开发 / 测试环境下通过受控替身数据源验证流程
4. 后续后端完成后，切换到真实数据源，不改页面层结构
5. 正式构建禁止接入 mock 数据源

### 10.1 推荐数据流

```text
Page
 -> Controller / Provider
   -> Service
     -> StubDataSource（开发/测试）
     -> ApiDataSource（预发/正式）
```

### 10.2 开发与测试阶段需要的受控数据

- 检测结果替身数据
- 历史记录替身数据
- 实时识别 overlay 替身数据
- 设置页配置替身数据

## 11. 重点页面设计建议

### 11.1 首页设计

建议布局：

- 顶部标题区
- 作品简介区
- 功能入口区
- 底部状态 / 版本信息

首页重点是：

- 不空
- 不像临时开发页
- 一眼能看懂作品在干什么
- 有明显视觉主次和作品感
- 首屏可加入轻量入场动画，提升产品进入时的完整感

### 11.2 结果页设计

建议布局：

- 原图区域 / 结果图区域
- 检测结果信息卡片
- 严重程度卡片
- 防治建议卡片
- 保存记录按钮

结果页必须做到：

- 信息清楚
- 层次明显
- 适合长期阅读和重复使用
- 图片、结果、建议三个区域分区明确
- 识别完成后信息分步呈现，避免结果突然堆满整个页面

### 11.3 实时识别页设计

建议使用 `Stack`：

- 底层：摄像头预览
- 中层：检测框 Overlay
- 顶层：状态栏 + 结果卡片

这个页面的重点不是花哨，而是：

- 稳定
- 清楚
- 有“实时识别”的观感
- 控件层不遮挡核心识别画面
- 检测框刷新和信息更新要平滑，不能频繁抖动

## 12. Material Design 风格建议

### 12.1 色彩建议

- 主色：绿色系，如 `Color(0xFF2E7D32)`
- 辅助色：浅绿色 / 青绿色
- 背景色：浅灰白
- 卡片色：纯白
- 错误色：红色
- 警告色：橙色

### 12.2 组件风格建议

- Button：圆角 + 统一高度
- Card：中等圆角 + 轻阴影
- 输入框：简洁边框
- AppBar：简洁，不要堆很多元素
- 图标：统一使用 Material Icons
- 弹窗 / BottomSheet / SnackBar 风格统一

### 12.3 文本层级建议

- 一级标题：页面标题
- 二级标题：模块标题
- 正文：识别说明、建议内容
- 辅助文字：时间、置信度、状态

### 12.4 视觉细节建议

- 首页可加入农业识别相关插画、渐变背景或品牌区块，避免过于朴素
- 卡片之间保持统一留白，避免页面拥挤
- 重点操作按钮使用高对比主色，次级操作降低视觉权重
- 结果页中的严重程度、置信度、建议内容可通过标签色和信息块强化识别效率
- 所有页面遵循统一栅格和边距规则，避免不同页面像不同项目拼接而成

### 12.5 跨平台交互规范

- 导航层级统一：启动页 -> 首页 -> 功能页 -> 结果详情页
- 返回行为统一：详情返回上一级，首页作为主入口不出现歧义返回
- 主要操作入口统一放在页面主要可视区域，不因平台变化而移动到完全不同位置
- 加载、错误、空数据、成功保存等反馈文案与反馈形式统一
- 仅在必须时处理平台差异，例如摄像头权限弹窗、Windows / Linux 窗口缩放、Web 文件选择
- 不在 iOS 单独设计一套路由逻辑，也不在 Web 单独设计另一套结果流程
- 桌面端操作仍以同一套页面流程为主，不额外拆分成独立业务版界面

### 12.6 动效规范

- 页面切换使用统一转场，不同功能页不要混用完全不同的动画风格
- 常规动效时长建议控制在 `180ms ~ 300ms`
- 卡片、列表、结果模块优先使用淡入、位移、缩放等轻量动画
- 状态切换优先使用 `AnimatedSwitcher`、`AnimatedContainer`、`Hero` 等原生能力
- 加载过程可使用骨架屏、渐隐渐现或轻量脉冲动画，不使用夸张长动画
- 实时识别页中 Overlay 更新要平滑，优先保证稳定帧率而非复杂特效
- 动效不可影响点击响应、滚动流畅度和识别页可读性

### 12.7 使用体验规范

- 点击按钮后立即提供视觉反馈，例如按压态、加载态、禁用态
- 异步操作必须体现处理中状态，避免用户误以为卡死
- 页面首次进入优先显示结构化骨架或占位，而不是长时间空白
- 空数据页需要给出说明和下一步操作建议
- 错误提示应说明问题并提供重试入口，避免只弹出模糊报错
- 桌面端与 Web 端补充鼠标悬停态、焦点态、滚轮滚动和快捷键支持
- 长列表、历史记录、设置页等页面要保证滚动顺畅和点击目标足够明确

## 17. 验收标准建议

为了满足生产环境标准，建议在文档中明确发布前验收标准：

### 17.1 页面验收

- 所有核心页面可正常跳转
- 页面文案统一中文
- 主题、色彩、按钮风格统一
- 空状态、加载态、失败态有基本占位
- 首页、结果页、实时识别页具备生产级视觉完成度
- 页面转场和模块入场动画风格统一，不突兀、不拖沓

### 17.2 交互验收

- 单图识别流程基于真实或测试环境接口可完整跑通
- 历史记录可新增、可查看详情
- 实时识别页可显示预览和受控 overlay，真实链路可平滑切换
- Android、iOS、Web、Windows、Linux 的主流程入口和操作顺序保持一致
- 点击、加载、保存、失败重试等关键操作具备明确且及时的反馈

### 17.3 架构验收

- 页面层不直接写网络请求
- 数据模型、service、controller 已拆分
- 开发替身数据源与正式 API 数据源可独立切换
- 路由与主题集中管理
- 公共组件已抽离，避免不同平台各自实现一套 UI 逻辑
- 对桌面端不单独复制业务层代码，平台差异控制在适配层
- 动画能力优先基于统一组件和统一规范实现，避免各页面各写一套
- `baseUrl`、超时、请求头、错误处理等网络配置集中管理

### 17.4 联调验收

- 前端可通过统一 `ApiClient` 发起 HTTP 请求，不在页面中直接拼接口
- 前后端按统一 JSON 结构联调，至少明确 `code`、`message`、`data` 三个字段
- 图片上传接口已预留 `multipart/form-data` 方案
- Java 后端常见返回异常，如 401、403、404、500，可被前端正确处理和提示
- 开发 / 测试环境可修改服务地址，正式环境通过安全配置方式下发

### 17.5 性能与体验验收

- 主要页面切换流畅，无明显掉帧、卡顿或白屏等待
- 实时识别页在展示假框和状态信息时保持稳定，不出现明显闪烁
- 长列表滚动、卡片展开、弹窗弹出等常见操作响应自然
- 低性能设备或桌面端窗口缩放时，动画仍保持可接受的顺滑度

### 17.6 测试与发布验收

- 关键业务逻辑具备单元测试
- 核心页面具备 Widget 测试
- 关键链路至少具备一条集成测试
- 发布构建通过格式化、静态检查、测试和多平台构建校验
- 版本号、变更记录、构建产物和回滚方案完整

### 17.7 安全与稳定性验收

- 正式环境不使用 mock 数据源
- 敏感配置、令牌和设备相关信息不明文硬编码
- 正式环境默认使用 HTTPS 或受控内网环境
- 崩溃采集、关键错误上报和关键日志链路可用
- 常见异常场景具备统一失败处理与恢复路径
- 不存在通过静默降级、假结果或默认成功值隐藏真实问题的实现

## 18. 结论

本项目 Flutter 前端应以“中文、Material 风格、模块解耦、可扩展、全平台一致性、优秀体验、生产可发布”为核心目标，采用 Feature-first 分层架构，优先搭建：

- 页面框架
- 路由体系
- 统一主题
- 真实接口与开发替身数据切换能力
- 摄像头预览页
- 结果展示页

在后端未完全就绪的阶段，前端可以通过受控替身数据源并行开发；但正式环境必须以真实服务、质量门禁、可观测性和发布规范为基线，无需在上线前再进行架构层面的推倒重来。
