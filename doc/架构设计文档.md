# 斛生 客户端架构重构设计文档

## 1. 文档目标

本文档用于回答两个问题：

1. 当前代码为什么会继续向屎山演化
2. 接下来应该按什么目标架构重构，才能在不打断现有功能的前提下收敛复杂度

本文档描述的是目标重构架构，不再只记录“现在代码放在哪里”，而是明确后续改造边界、模块归属和迁移顺序。

## 2. 现状诊断

当前项目已经具备可运行基线，但维护成本正在快速上升，主要问题不是单个页面丑，而是架构边界失真。

### 2.1 Feature 边界失真

典型问题：

- 重构前 `lib/features/settings/device_state_repository.dart` 实际承担的是设备状态与 LED 控制能力，但被首页、值守页、设置页共同依赖
- `home`、`realtime`、`settings` 三个页面都在直接感知“设备状态”这条业务链路，却没有一个独立的 `device` 业务模块
- `about`、`video` 等页面已经存在独立 feature，但 `架构设计文档` 之前没有完整反映最新模块边界

影响：

- 业务所有权不清楚
- 一个能力改动会跨多个 feature 扩散
- “哪个目录该放什么”只能靠记忆和习惯维持

### 2.2 领域模型混入展示逻辑

典型问题：

- 重构前 `lib/shared/models/device_state_info.dart` 同时承担：
  - 接口 JSON 解析
  - 设备状态实体
  - 用户文案计算
  - 指标格式化
  - 告警等级映射

这意味着一个“纯数据对象”已经知道：

- 页面要显示什么标题
- 页面要显示什么说明
- 指标要怎么拼接单位

影响：

- UI 文案改动会直接污染领域模型
- 同一模型被不同页面复用时，表现形式很难分层
- 测试不得不混着测数据和展示规则

### 2.3 超大页面文件与超大壳层

重构启动前，这些文件已经明显超过项目自己的文件规模约束：

- `lib/features/video/video_page.dart` 1082 行
- `lib/features/about/about_page.dart` 776 行
- `lib/app/widgets/app_workspace_scaffold.dart` 643 行
- `lib/features/home/widgets/home_header_card.dart` 511 行
- `lib/features/realtime/widgets/realtime_monitor_hero.dart` 484 行

问题本质不是“行数大”，而是：

- 页面文件同时定义页面容器、Section、局部组件、绘制器和交互逻辑
- 一个视觉改动会让整页文件继续膨胀
- 复用边界靠大量私有 Widget 硬撑，而不是靠模块职责划分

截至 2026-03-12，这几处入口文件都已经拆回到可维护尺寸，超大文件问题主要转移为局部视觉组件的继续细化。

### 2.4 页面直接承接业务生命周期

典型问题：

- `HomePage` 自己持有定时刷新 Timer
- `RealtimeDetectPage` 在页面 `initState/dispose` 中直接启动和停止监控
- 页面自己负责拼接当前用户名称

影响：

- 生命周期策略分散在页面里
- 后续如果出现更多入口共享状态，页面层会越来越像控制器层
- 页面测试需要知道业务时序细节

### 2.5 共享状态缺少统一派生层

当前多个页面反复做同样的转换：

- `currentUser = displayName ?? account ?? '--'`
- 设备名称为空时的兜底判断
- 同一业务状态在不同页面里各自转换为不同展示块

影响：

- 重复逻辑多
- 改一处展示口径需要全局扫页面
- 页面 build 中混入了大量“胶水代码”

### 2.6 历史残留没有完全清场

重构前仓库仍保留空目录：

- `lib/features/detect`
- `lib/features/history`
- `lib/features/result`

这类残留会持续误导后续开发者，导致“是不是以后还会恢复”“能不能继续往里面塞代码”这类错误预期。

## 3. 重构目标

这次架构设计不追求教科书式纯洁，而是追求三件事：

1. 业务能力有明确归属
2. 页面只负责组织展示，不直接承担核心业务编排
3. 允许增量迁移，不要求一次性推倒重来

## 4. 目标架构

目标采用：

`feature-first + 内部分层 + 共享业务模块前置`

核心思想：

- 继续保留 `feature-first`
- 但每个核心业务 feature 内部拆出 `application / domain / infrastructure / presentation`
- 所有被多个页面共享的业务能力，必须抽成独立业务 feature，而不是挂在某个页面 feature 下面

### 4.1 当前已落地状态

当前仓库已经完成共享业务归位和命名收敛：

- `device` 共享业务入口已经统一收敛到 `lib/features/device/application/device_runtime_providers.dart`
- 设备状态实体已经收敛到 `lib/features/device/domain/device_status.dart`
- 设备运行时仓储契约已经收敛到 `lib/features/device/domain/device_runtime_repository.dart`
- 设备远端仓储实现已经收敛到 `lib/features/device/infrastructure/device_remote_runtime_repository.dart`
- LED 操作结果已经独立建模到 `lib/features/device/domain/led_operation_receipt.dart`
- 当前用户显示名称已经统一收敛到 `lib/features/auth/application/current_user_label_provider.dart`

页面层胶水代码的第二阶段也已经完成主体收敛：

- 设备展示派生已经下沉到 `lib/features/device/application/device_status_view_data.dart`
- 首页自动刷新已经下沉到 `lib/features/home/application/home_overview_device_status_provider.dart`
- 值守页监控启动和 LED 延迟回刷已经收回 `RealtimeDetectController`
- 首页 Hero、首页设备快照、值守 Hero、帮助页、视频页、工作台壳层和登录表单都已经拆成页面容器 + 局部组件文件

当前剩余工作不再是共享业务归属问题，而是按需要继续细化局部视觉组件边界。

## 6. 模块归属设计

### 6.1 `workspace`

职责：

- 工作台导航壳层
- 顶部标题仓
- 桌面端 `NavigationRail`
- 移动端 `NavigationBar`

迁移目标：

- 从 `lib/app/widgets/app_workspace_scaffold.dart` 拆出多个工作台子组件文件
- `app` 层只负责组装，不再长期承载复杂 UI 壳层细节

当前实现：

- `lib/app/widgets/app_workspace_scaffold.dart`
- `lib/app/widgets/workspace/workspace_content_pane.dart`
- `lib/app/widgets/workspace/workspace_header_card.dart`
- `lib/app/widgets/workspace/workspace_rail_pane.dart`
- `lib/app/widgets/workspace/workspace_bottom_navigation.dart`
- `lib/app/widgets/workspace/workspace_navigation.dart`

### 6.2 `auth`

职责：

- 登录态恢复
- 登录 / 注册 / 退出
- 当前用户派生信息

分层建议：

- `domain`: `AuthSession`、`AuthUser`、`AuthRepository`
- `application`: `AuthController`、`currentUserLabelProvider`
- `infrastructure`: `RealAuthRepository`、`MockAuthRepository`、存储适配
- `presentation`: `LoginPage`、表单和认证壳层

### 6.3 `device`

职责：

- 设备状态查询
- LED 控制
- 设备状态轮询
- 跨页面共享的设备摘要与展示模型

这是当前最应该新增的 owning feature。

分层建议：

- `domain/entities`: `DeviceStatus`
- `domain/value_objects`: `DeviceAlertLevel`
- `domain/repositories`: `DeviceRuntimeRepository`
- `application`: `DeviceMonitorController`、`deviceStatusProvider`、`deviceSummaryProvider`
- `infrastructure`: `DeviceRuntimeRepositoryImpl`、`DeviceRemoteDataSource`、DTO、Mapper
- `presentation`: 与设备能力强绑定但可跨页面复用的 Section 和 Presenter

强制规则：

- 首页、值守页、设置页都只能依赖 `device` 模块暴露的应用层 Provider
- 不允许再从 `settings` 下引用设备仓储

### 6.4 `home`

职责：

- 只组织总览页布局
- 消费 `auth` 和 `device` 的派生状态
- 不直接持有轮询和仓储

页面只应该知道：

- 当前用户是谁
- 当前设备摘要是什么
- 轨道入口怎么排

### 6.5 `realtime`

职责：

- 只组织值守页布局和交互
- 依赖 `device` 模块的监控状态与控制动作

迁移目标：

- `RealtimeDetectPage` 不再直接决定监控生命周期
- 页面只负责触发应用层提供的动作

### 6.6 `video`

职责：

- 视频流列表获取
- 视频查看状态派生
- 视频卡片展示

分层建议：

- `domain`: `VideoStream`
- `application`: `VideoCenterController` 或纯查询 Provider
- `infrastructure`: `VideoStreamRepositoryImpl`、远端 DTO、Mapper
- `presentation`: `VideoPage`、Hero、Gallery、StreamCard、Tips

### 6.7 `settings`

职责：

- 本机偏好
- 记住账号
- 恢复默认
- 退出登录入口聚合

强制规则：

- `settings` 不拥有设备状态仓储
- `settings` 页面里的设备信息只能来自 `device` 模块

### 6.8 `about`

职责：

- 软件使用说明
- 页面查看顺序引导

说明：

- 这是纯 presentation feature
- 不应该承载新的业务状态或仓储

## 7. 分层边界

### 7.1 Presentation 层

职责：

- 页面和区块布局
- 用户交互事件转发
- 展示状态渲染

禁止：

- 直接发请求
- 直接解析 JSON
- 直接计算复杂业务口径

### 7.2 Application 层

职责：

- 用 Riverpod 组织状态
- 编排 UseCase
- 管理轮询、刷新、提交、错误态
- 暴露给页面的 view state / derived state

建议规则：

- 页面长生命周期行为放这里
- 跨页面复用的派生状态放这里

### 7.3 Domain 层

职责：

- 纯业务实体
- 仓储接口
- 不依赖 Flutter

强制规则：

- 不写用户文案
- 不关心颜色、按钮和页面布局

### 7.4 Infrastructure 层

职责：

- DTO
- Mapper
- Remote / Local Data Source
- Repository 实现

强制规则：

- 所有网络访问都收敛到这里
- `ApiClient` 只作为底层能力，不允许直接暴露给页面

## 8. 数据流设计

### 8.1 查询链路

`Page -> ScreenSection -> Application Provider -> UseCase / Controller -> Repository Contract -> Repository Impl -> Remote Data Source -> ApiClient`

### 8.2 提交链路

`User Action -> Presentation Callback -> Application Action -> Repository Contract -> Repository Impl -> ApiClient -> Result / Failure -> Application State -> UI`

### 8.3 展示口径链路

`Domain Entity -> Presenter / ViewData Mapper -> UI Widget`

这条链路的关键点：

- 用户文案不回到 Domain Entity
- 页面不自己拼所有展示字段

- 把 `DeviceRuntimeRepository` 从 `settings` 中迁出

## 10. 增量迁移策略

### Phase 1

先抽共享业务，不先动全部页面：

- 引入 `device` 模块
- 保持现有页面结构可跑
- 通过适配层兼容旧调用

### Phase 2

再拆超大文件：

- 每次只拆一个页面或一个壳层
- 不在同一提交里同时大改业务和视觉

### Phase 3

统一派生状态：

- `currentUserLabelProvider`
- `deviceSummaryProvider`
- `deviceMetricsViewDataProvider`

### Phase 4

清理遗留目录与旧命名：

- 删除空模块
- 修正导入路径
- 更新文档和测试命名

## 11. 守护规则

从本次重构开始，新增代码必须满足：

1. 业务实体不写 UI 文案
2. 共享业务能力不挂在页面 feature 下
3. 页面只依赖应用层 Provider，不依赖仓储实现
4. 单文件超过 400 行必须拆分
5. 单个页面 build 超过 120 行必须抽 Section
6. 新功能必须先明确 owning feature，再开始写代码

## 12. 完成判定

满足以下条件，才算这次架构重构完成：

- `device` 业务从 `settings` 中完全解耦
- 当前大页面全部拆成页面容器 + section + widget
- `shared` 中不再堆业务实体
- 页面层不再直接持有核心业务生命周期
- 目录结构和文档一致
- 测试覆盖新的模块边界和核心应用层状态
