# 石斛病虫害识别项目 Flutter 前端架构设计文档

## 1. 文档目标

本文档用于说明石斛病虫害识别项目 Flutter 前端的当前架构基线、模块边界、目录结构、依赖策略和平台适配方式，作为后续继续开发和审查代码的统一依据。

本文档不再按“从 0 开始规划”的口径描述，而是以当前仓库中的真实代码结构为准。

## 2. 当前架构建设目标

当前前端基线的目标分为两层：

1. 先交付一个可运行、可扩展的单图识别主链路
2. 在不推翻现有页面和状态流的前提下，为真实接口、实时识别和全平台发布继续扩展

因此当前架构必须同时满足：

- 页面可运行
- 路由完整
- 状态可管理
- 仓储可替换
- 平台差异可收敛
- 文档和代码能够同步演进

## 3. 平台范围与策略

### 3.1 目标平台

- Android
- iOS
- macOS
- Web
- Windows
- Linux
- OpenHarmony / 鸿蒙

### 3.2 当前平台策略

- 业务代码尽量共用，不按平台复制页面
- 文件选择、权限、运行时平台判断等差异收敛到适配层
- `macOS` 已纳入原生平台工程
- `OpenHarmony / 鸿蒙` 已纳入平台枚举、运行时适配和平台工程目录
- `OpenHarmony / 鸿蒙` 的实际构建依赖本地 OHOS 工具链与签名，不应在文档中误写成“任何机器直接可构建”

## 4. 当前功能模块边界

当前代码包含以下业务模块：

- `splash`：应用初始化与首页跳转
- `home`：主入口与基线说明
- `detect`：单图识别主链路
- `result`：识别结果展示和历史记录保存
- `history`：本地历史记录浏览、删除、清空
- `settings`：环境、平台、版本和本地配置管理
- `about`：项目定位与技术基线说明

补充说明：

- `realtime_detect` 当前尚未形成独立功能模块，只有路由占位入口
- 网络层已经存在 `ApiClient` 和 `ApiException`，但单图识别还未切换到真实后端仓储

## 7. 状态管理策略

当前状态管理统一使用 `flutter_riverpod`，不再保留 `Provider` 与 `Riverpod` 二选一的模糊建议。

当前模式如下：

- 全局根节点通过 `ProviderScope` 挂载
- 路由通过 `appRouterProvider` 暴露
- 单图识别使用 `NotifierProvider`
- 历史记录和设置使用 `AsyncNotifierProvider`
- 版本信息和本地仓储通过 `FutureProvider` 暴露

当前核心 Provider：

- `appRouterProvider`
- `detectControllerProvider`
- `historyRepositoryProvider`
- `historyControllerProvider`
- `settingsControllerProvider`
- `packageInfoProvider`
- `envConfigProvider`

## 8. 当前数据流设计

### 8.1 启动链路

`SplashPage -> SettingsController -> LocalStorage`

同时：

`SplashPage -> packageInfoProvider -> PackageInfo / OHOS 兜底版本信息`

### 8.2 单图识别链路

`DetectPage -> DetectController -> DetectRepository -> RealDetectRepository / MockDetectRepository -> ResultPage`

说明：

- 当前 `DetectRepository` 默认走真实接口仓储
- 开发 / 测试环境可通过 `USE_MOCK_DETECT=true` 显式切回受控 mock
- 页面层已经不直接依赖具体数据源
- 页面层和结果页契约不依赖具体实现切换

### 8.3 历史记录链路

`ResultPage -> HistoryController.saveRecord -> HistoryRepository -> LocalStorage`

`HistoryPage -> HistoryController -> HistoryRepository -> LocalStorage`

说明：

- 当前历史记录为本地真实持久化
- 历史记录数据结构使用 `HistoryRecord`

### 8.4 设置链路

`SettingsPage -> SettingsController -> LocalStorage`

说明：

- 当前设置项以 `AppSettings` 持久化
- `baseUrl` 编辑权限由 `EnvConfig.allowRiskySettings` 控制

## 9. 数据源策略

当前数据源策略必须明确区分：

- 单图识别：默认真实接口，开发 / 测试环境可切回 mock
- 历史记录：本地持久化
- 设置：本地持久化
- 实时识别：仅占位，未接入

禁止继续使用以下不准确表述：

- “历史记录当前用 mock 数据”
- “设置页当前用 mock 配置”
- “实时识别当前已有假框演示”

## 10. 网络层策略

当前网络层已有基础封装：

- `ApiClient`
- `ApiException`

当前能力：

- 统一 `GET / POST / multipart` JSON 返回
- 统一 `DioException -> ApiException` 映射
- 自动注入 `Accept` 和 `X-Platform`

当前未完成：

- 通用 `ApiResponse<T>` 类型封装
- 单图识别真实接口仓储实现
- 实时识别接口仓储实现

## 11. 平台适配策略

当前平台适配集中在：

- `platform_utils.dart`
- `platform_runtime_io.dart`
- `platform_runtime_stub.dart`
- 自适应图片渲染组件

关键策略：

- `PlatformType` 已包含 `android / ios / macos / web / windows / linux / ohos`
- `tryPlatformTypeFromValue()` 接受 `openharmony`、`harmonyos`、`ohos` 并统一解析为 `ohos`
- `APP_PLATFORM_OVERRIDE=ohos` 可用于编译期平台覆盖
- `package_info_plus` 在 `OpenHarmony / 鸿蒙` 下使用代码兜底，避免插件缺失导致初始化失败

## 12. 依赖与插件基线

当前核心依赖为：

- `flutter_riverpod`
- `go_router`
- `dio`
- `image_picker`
- `shared_preferences`

- `image_picker` 和 `shared_preferences` 当前使用仓库内置的 OpenHarmony 适配包路径
- 这属于平台兼容策略，不应在文档中再写成“待选型”
