# 斛生 Flutter 代码规范、注释规范与开发规范

## 1. 文档目标

本文档用于统一斛生 Flutter 客户端的工程标准，覆盖：

- 代码规范
- 注释规范
- 日常开发规范
- 测试与提交流程
- 发布前质量要求

本文档适用于 Android、iOS、macOS、Web、Windows、Linux、OpenHarmony / 鸿蒙 全平台 Flutter 前端开发，并以生产环境标准为基线。

## 2. 基本原则

所有代码与开发行为必须遵循以下原则：

- 可读性优先于炫技
- 一致性优先于个人习惯
- 可测试性优先于临时可跑
- 可维护性优先于一次性写完
- 正式环境稳定性优先于开发便利性

补充要求：

- 代码标识符统一使用英文
- 界面文案统一使用中文，并集中管理
- 不允许把临时调试代码、临时注释块、无来源的 TODO 带入正式分支

## 3. 代码规范

### 3.1 通用规范

- 必须启用 Dart Null Safety
- 必须通过 `dart format` 格式化
- 必须通过 `flutter analyze`
- 不允许提交存在明显警告、未使用变量、未使用导入的代码
- 不允许在业务代码中长期保留 `print`、`debugPrint`
- 正式代码统一通过日志组件输出日志

建议命令：

```bash
dart format .
flutter analyze
flutter test
```

### 3.2 命名规范

- 文件名使用 `snake_case`
- 文件夹名使用 `snake_case`
- 类名、枚举名、类型名使用 `UpperCamelCase`
- 变量名、方法名、参数名使用 `lowerCamelCase`
- 私有成员使用前缀 `_`
- 常量名使用 `lowerCamelCase`
- 路由名、Provider 名、Repository 名、Service 名应体现职责，不允许使用模糊命名

示例：

- 正确：`realtime_monitor_top_bar.dart`
- 正确：`SettingsOverviewCard`
- 正确：`realtimeDetectControllerProvider`
- 错误：`test.dart`
- 错误：`DataManager`
- 错误：`doThing()`

### 3.3 文件与目录规范

- 文件结构必须服从项目架构文档，不允许绕过分层随意放置
- 页面、状态、服务、模型、组件必须按职责拆分
- 单个文件建议不超过 400 行
- 单个函数建议不超过 50 行
- 单个 `build` 方法建议不超过 120 行
- 超过阈值时优先拆分组件、抽取方法或重构状态

禁止事项：

- 在页面文件中直接写网络请求
- 在页面文件中直接解析 JSON
- 在多个页面复制相同布局和相同逻辑

### 3.4 Import 规范

- 优先使用包导入或项目内统一导入风格，不混用相对路径和包路径
- Import 顺序保持一致：Dart SDK、第三方包、项目内包
- 不允许保留未使用 Import
- 不允许通过一个“大而全”文件隐藏真实依赖关系

### 3.5 Widget 规范

- Widget 只负责展示和用户交互，不承担复杂业务决策
- 可复用 UI 必须抽取为独立 Widget
- 纯展示组件优先使用 `StatelessWidget`
- 局部短期状态可使用 `ConsumerStatefulWidget` 或受控局部状态
- 跨页面或可复用状态必须进入统一状态管理层
- 能使用 `const` 时必须使用 `const`
- 顶层工作台页面统一通过 `NavigationRail / NavigationBar` 切换，不在一级模块间叠加返回栈

推荐做法：

- 一个页面由页面容器、功能区块、通用组件组成
- 复杂列表项、结果卡片、设置项单独拆 Widget
- 避免在 `build` 中写过多数据处理逻辑

### 3.6 状态管理规范

- 生产项目默认使用 `flutter_riverpod`
- 页面不直接持有核心业务状态
- 状态变更必须通过 Provider、Notifier、Controller 或 UseCase 触发
- 异步状态统一管理 `loading`、`data`、`error`
- 不允许把多个互不相关的状态堆在一个超大 Provider 中

推荐规则：

- 页面状态和业务状态分离
- 短生命周期 UI 状态与长期业务状态分离
- Provider 命名要体现职责，例如 `settingsControllerProvider`

### 3.7 模型与数据规范

- 接口模型、领域模型、持久化模型职责明确
- JSON 模型统一使用 `json_serializable` 或 `freezed`
- 不手写重复 JSON 映射代码，除非有明确性能或兼容性理由
- 所有外部数据进入业务层前必须完成校验和兜底
- 不允许在 UI 层拼接复杂业务字段

推荐做法：

- `ApiResponse<T>` 统一处理后端基础包裹结构
- 使用 Mapper 或扩展方法做模型转换
- 对可空字段、枚举值异常、空列表等场景做兼容

### 3.8 网络与错误处理规范

- 所有网络请求统一走 `ApiClient`
- 请求超时、连接失败、401、403、404、500 必须统一处理
- 错误提示要面向用户，错误日志要面向排查
- 不允许吞异常
- 不允许在 catch 中只写空代码块
- 重试策略、降级策略、空数据策略必须有明确边界
- 退出登录、恢复默认设置、清除本机记住账号等危险动作必须二次确认

推荐做法：

- 在网络层完成基础异常归一化
- 在业务层处理业务语义错误
- 在页面层负责展示错误状态和恢复入口

### 3.9 本地存储与安全规范

- 普通配置与轻量缓存可使用 `shared_preferences`
- 敏感信息必须使用 `flutter_secure_storage` 或同等级方案
- 不允许把令牌、密钥、正式环境地址硬编码到仓库
- 本地缓存要有生命周期和清理策略
- 删除缓存、清空记录等危险操作必须经过确认

### 3.10 动画与性能规范

- 动画服务于状态表达，不做无意义装饰
- 常规动效时长建议在 `180ms ~ 300ms`
- 实时识别页优先保证稳定帧率
- 长列表必须关注滚动性能
- 不在 `build` 中进行高频重计算
- 不在高频刷新链路中创建大量临时对象

### 3.11 平台适配规范

- 平台差异必须收敛到适配层，不允许散落在各页面
- Android、iOS、macOS、Web、Windows、Linux、OpenHarmony / 鸿蒙 主流程必须一致
- 平台专属逻辑要有统一封装
- 权限申请、摄像头初始化、文件选择等能力应通过抽象接口接入
- 不允许为不同平台复制一套完整业务页面

### 3.12 测试规范

- 纯业务逻辑必须有单元测试
- 核心页面必须有 Widget 测试
- 关键链路必须至少有一条集成测试
- Bug 修复应优先补对应测试
- 替身数据源可用于测试，但正式构建必须关闭

最低要求：

- 新增业务逻辑有对应测试
- 新增异常分支有对应测试
- 提交前至少本地通过 `flutter test`

## 4. 注释规范

### 4.1 注释原则

- 注释统一使用中文；必要的类名、方法名、协议名、平台名可保留英文标识，并使用反引号包裹
- 注释解释“为什么”，不是重复“做了什么”
- 能通过命名表达清楚的内容，不再额外写注释
- 注释必须和代码保持同步
- 过期注释比没有注释更糟

### 4.2 必须写注释的场景

- 对外暴露的公共类、公共方法、公共 Provider
- 对外暴露的公共枚举、公共模型及关键字段
- 复杂业务规则
- 关键边界条件
- 跨平台兼容处理
- 性能优化或规避框架限制的代码
- 容易误用的能力入口
- 文档定义发生变化后，与之对应的实现入口

### 4.3 DartDoc 规范

对外可复用的类型和方法，优先使用 `///` DartDoc。

补充要求：

- 文档中定义过职责的页面、Provider、Repository、模型、路由参数、平台适配入口，必须有对应 DartDoc
- 代码行为变更后，DartDoc 必须同步更新，不能保留旧描述
- 文档新增字段、枚举或状态后，代码中的公共模型和公共枚举注释必须同步补齐

示例：

```dart
/// 认证仓储统一入口。
///
/// 在开发和测试环境中可切换到本地联调替身，
/// 在正式环境中固定使用真实会话接口实现。
abstract class AuthRepository {
  Future<AuthSession> login({
    required String username,
    required String password,
  });
}
```

### 4.4 行内注释规范

行内注释仅用于解释：

- 业务原因
- 技术限制
- 兼容性处理
- 非直观实现

示例：

```dart
// 摄像头返回的是预览坐标，先归一化再映射到屏幕坐标，
// 否则桌面端窗口缩放后检测框会偏移。
final rect = mapPreviewRectToScreenRect(rawRect, previewSize, screenSize);
```

### 4.5 TODO / FIXME 规范

- 不允许写无责任人、无背景的 TODO
- TODO 必须说明原因和后续动作
- FIXME 只用于确认存在缺陷且必须后续修复的问题

推荐格式：

```dart
// TODO(2026-03-10, settings): Windows 端完成真机构建后补齐服务地址编辑验证。
// FIXME(2026-03-10, auth): 某平台若出现 Cookie 丢失，需要补抓会话恢复日志。
```

### 4.6 禁止的注释

- 注释掉的大段旧代码
- 与代码含义重复的废话注释
- 情绪化注释
- “临时先这样”“不知道为什么但能跑”这类不可维护注释

错误示例：

```dart
// 给变量赋值
final name = user.name;

// 先这样，别动
```

## 5. 开发规范

### 5.1 分支规范

推荐分支模型：

- `main`：正式发布分支
- `develop`：日常集成分支
- `feature/*`：功能开发分支
- `fix/*`：问题修复分支
- `hotfix/*`：线上紧急修复分支

要求：

- 不直接在 `main` 上开发
- 一个分支只做一类事情
- 大功能拆分为多个可审查的小分支

### 5.2 任务开发流程

标准流程：

1. 阅读相关代码、文档、接口契约和现有测试
2. 明确需求、验收条件、影响范围与平台差异
3. 创建功能分支
4. 开发代码并补测试
5. 先完成一次自审
6. 同步修改受影响文档
7. 同步补齐受影响注释
8. 执行本地自检
9. 使用中文提交代码并发起评审
10. 修复评审意见
11. 合并并验证集成结果

补充要求：

- 不允许跳过“阅读代码”“自审代码”“同步文档”“同步注释”四个步骤
- 文档未更新完成、注释未补齐完成，不允许进入提交流程

### 5.3 提交规范

提交信息建议使用 Conventional Commits：

- `feat:` 新功能
- `fix:` 缺陷修复
- `refactor:` 重构
- `test:` 测试
- `docs:` 文档
- `chore:` 工程或依赖调整

示例：

```text
feat: 完成主控台与设置页统一工作台壳层
fix: 修复健康检查失败时的错误提示
docs: 补充 Flutter 开发规范与工作流要求
```

要求：

- 一次提交只表达一个明确意图
- 提交信息不能写成 `update`、`modify`、`test`
- 提交标题和必要说明统一使用中文
- 提交说明必须自然、直接、工程化，不允许出现 AI 痕迹
- 不允许出现“AI 生成”“模型建议”“智能优化”“由 AI 完成”等描述
- 提交前确保代码可编译、可分析、可测试

### 5.4 代码评审规范

评审重点：

- 是否满足需求
- 是否破坏现有行为
- 是否符合架构分层
- 是否存在性能、稳定性、安全问题
- 是否补充必要测试
- 是否引入重复代码

评审要求：

- 不接受“先合再说”的高风险代码
- 不接受没有验证的核心链路修改
- 不接受把架构问题留给后续统一重构

### 5.5 本地检查规范

开发者在提交前至少完成：

```bash
dart format .
flutter analyze
flutter test
```

如使用代码生成，还应执行：

```bash
dart run build_runner build --delete-conflicting-outputs
```

### 5.6 依赖管理规范

- 新增依赖必须说明用途、平台兼容性和替代方案
- 不引入长期无人维护的包
- 不为小问题引入重量级依赖
- 依赖版本升级要评估破坏性变更
- 生成代码相关依赖必须固定在团队可接受的版本范围

### 5.7 配置与环境规范

- 开发、测试、正式环境严格隔离；当前环境值固定为 `development / staging / production`
- 环境差异通过配置层控制，不通过代码里到处 `if` 判断
- 正式环境默认关闭调试日志、替身数据入口和高风险配置入口
- 构建产物必须标记版本号、构建号和环境信息

### 5.8 发布规范

发布前必须确认：

- 版本号已更新
- 变更记录已整理
- 关键链路已回归
- 多平台构建通过
- 崩溃采集和日志链路可用
- 回滚方案明确

发布执行要求：

- 主分支只允许提交源码、必要工程文件和项目文档，不允许夹带编译产物和本地开发配置
- 标准分发统一通过 GitHub Actions `发布安装包` 工作流完成
- 正式发布优先使用 `v*` 标签触发；补发场景允许手动执行工作流并显式填写标签
- 本地手工发包统一输出到仓库根目录 `release/`，仅作为待上传分发物目录，不纳入主分支版本管理
- GitHub 官方 runner 只上传面向分发的安装类或便携类产物，不上传原始 build 目录、Web 静态资源和校验附件
- GitHub 官方 runner 负责 Android APK、Linux deb/rpm/pacman/AppImage/Flatpak/便携包、Windows 安装器、macOS dmg 和未签名 iOS IPA 的构建
- 若个别平台在工作流中失败，必须保证已成功的平台产物仍可继续汇总并上传 Release，不允许单个平台失败阻断全部已成功产物
- OpenHarmony / 鸿蒙 发布必须使用带 `self-hosted`、`linux`、`ohos` 标签的自托管 runner
- iOS 在仓库工作流中默认产出未签名 IPA；如需真机直接安装或外部分发，必须再补签名流程

补充说明：

- Linux 冷门架构如 `loong64`、`riscv64` 当前不能依赖官方 Flutter runner 直接交叉构建
- 如需分发此类架构，必须先在对应架构机器上产出原生 Linux bundle，再调用仓库内统一打包脚本补系统包封装
- Linux 桌面端如调整原生 CMake 配置，必须保证 `flutter build linux --release` 在默认启用硬化策略的宿主机上可直接通过，不允许依赖临时环境变量补 `PIE` 相关参数
- OpenHarmony / 鸿蒙 构建必须统一走 `tool/build_ohos_release.sh`，由脚本在临时工作区生成兼容 `flutter-ohos 3.35.x` 的依赖图，不允许在主工作区手改 `pubspec.lock`、`.dart_tool/` 或 `.flutter-plugins-dependencies`
- `ohos/package-lock.json`、`.vscode/` 等本地生成配置不得进入主分支

### 5.9 文档同步规范

以下变更必须同步更新文档：

- 架构分层调整
- 接口契约变化
- 配置项变化
- 发布流程变化
- 关键业务流程变化

补充要求：

- 文档一旦修改，必须反查对应代码注释是否仍然准确
- 与文档对应的公共页面、公共 Provider、公共 Repository、公共模型、公共枚举，必须同步补充或更新注释
- 不允许出现“文档已更新但代码注释还是旧逻辑”的情况

## 6. 禁止事项

- 禁止把业务逻辑直接塞进页面 `build`
- 禁止在正式代码中保留临时代码路径
- 禁止提交未格式化代码
- 禁止绕过统一网络层直接请求接口
- 禁止把异常静默吞掉
- 禁止在正式环境开启替身数据源
- 禁止在没有测试和验证的情况下修改核心链路
- 禁止将敏感信息提交到仓库

## 7. 开发自检清单

提交前至少自查以下内容：

- 命名是否清晰
- 文件是否按分层放置
- 是否有重复逻辑可抽取
- 是否存在无意义注释或过期注释
- 文档是否已同步更新
- 文档对应注释是否已同步补齐
- 是否补充必要测试
- 是否通过格式化和静态检查
- 是否处理了空状态、失败态和权限异常
- 是否影响多平台行为一致性
- 是否引入安全风险或正式环境配置风险

## 8. 结论

本项目开发必须以生产环境标准为前提，代码要做到：

- 结构清晰
- 命名统一
- 注释克制且有效
- 状态与数据边界明确
- 测试、日志、发布流程完整

后续所有 Flutter 前端代码、注释、提交流程和评审流程，均以本文档为统一规范。
